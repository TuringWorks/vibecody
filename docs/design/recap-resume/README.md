# Recap & Resume — Design Index

**Status:** Draft · 2026-04-26
**Scope:** vibecli (daemon + REPL/TUI), vibeui (Tauri desktop), vibemobile (Flutter), vibewatch (watchOS + Wear OS)
**Owner:** TBD

---

## What this is

A unified design for two related capabilities across every VibeCody client:

- **Recap** — a structured, on-demand summary of a unit of work (a chat session, a background job, or a diffcomplete edit chain). Generated by the daemon, cached durably, surfaced consistently in every client.
- **Resume** — the ability to pick up *where you left off* on any unit of work, on any client, without re-discovering context.

These are two halves of the same primitive: a recap is the artifact that lets a resume happen cheaply (the user, the model, and the next client all read the same summary).

## Three scopes, one shape

| Scope | Unit of work | Doc |
|---|---|---|
| **Session** | A chat / agent conversation (one or more turns) | [`01-session.md`](./01-session.md) |
| **Job** | A background agent run, Counsel deliberation, or Arena comparison | [`02-job.md`](./02-job.md) |
| **Diffcomplete** | A regenerate-with-refinement chain on one selection | [`03-diffcomplete.md`](./03-diffcomplete.md) |

Each scope owns its data model and UX, but they share the same daemon primitives, RPC contract, storage rules, and security posture — defined here.

## Goals

1. **One source of truth.** The vibecli daemon owns recap generation and durable storage; every client renders what the daemon emits. Clients never compute their own recap from raw history. (Aligns with [AGENTS.md](../../../AGENTS.md): the daemon is the authoritative protocol surface.)
2. **Cheap to resume on any device.** Opening a session on your watch should not require streaming the full transcript — the recap is enough to get oriented; on-demand fetch fills in detail.
3. **No surprise capture.** Recap is generated on explicit triggers (session end, job completion, user request) — not silently in the background. Diffcomplete remains user-directed (see patent-distance constraint below).
4. **Resume preserves the chain, not just the last state.** Branching, prior diffs, prior reasoning — all reachable from the recap.

## Non-goals

- **Not** a memory / cognitive system. Recap is per-unit-of-work; long-term memory across units lives in `/memory/*` (already in `serve.rs`) and the local `MEMORY.md` projection. Recap can *feed* memory, but does not replace it.
- **Not** a replacement for `git` checkpoints. Code state is still snapshotted via `create_checkpoint` (git stash). Recap covers conversation/job/diff state, not file state.
- **Not** an analytics or telemetry surface. Recaps are user-private; never sent off-device unless the user explicitly shares them.

## Cross-cutting data model

A single normalized shape covers all three scopes. The `kind` field disambiguates.

```rust
// vibecli/vibecli-cli/src/recap.rs (new)
pub struct Recap {
    pub id: RecapId,                       // ULID, sortable by creation time
    pub kind: RecapKind,                   // Session | Job | DiffChain
    pub subject_id: String,                // session_id | job_id | diff_chain_id
    pub workspace: Option<PathBuf>,        // owning workspace, if any
    pub generated_at: DateTime<Utc>,
    pub generator: RecapGenerator,         // Heuristic | Llm { provider, model } | UserEdited
    pub headline: String,                  // <= 80 chars, single line, plain prose
    pub bullets: Vec<String>,              // 3–7 short bullets, what happened
    pub next_actions: Vec<String>,         // 0–3 imperative-form follow-ups
    pub artifacts: Vec<RecapArtifact>,     // file paths touched, diffs produced, jobs spawned
    pub resume_hint: Option<ResumeHint>,   // structured handoff to a resume call
    pub token_usage: Option<TokenUsage>,
    pub schema_version: u16,               // start at 1
}

pub enum RecapKind { Session, Job, DiffChain }

pub enum RecapGenerator {
    Heuristic,                              // template-fill, no LLM
    Llm { provider: String, model: String },
    UserEdited,                             // human edited an LLM/heuristic recap
}

pub struct RecapArtifact {
    pub kind: ArtifactKind,                 // File | Diff | Job | Url
    pub label: String,
    pub locator: String,                    // path, diff-id, job-id, url
}

pub struct ResumeHint {
    pub target: ResumeTarget,               // Session(id) | Job(id) | DiffChain(id)
    pub from_message: Option<MessageId>,    // session resume cursor
    pub from_step: Option<u32>,             // job step cursor
    pub from_diff_index: Option<u32>,       // diff-chain cursor
    pub seed_instruction: Option<String>,   // pre-fill the next prompt
}
```

## Cross-cutting RPC surface

All routes mount under the existing daemon (`serve.rs`) with the standard Bearer token. Three families:

```
POST /v1/recap                      Generate or fetch a recap (idempotent on subject_id+kind)
GET  /v1/recap/:id                  Fetch a stored recap
GET  /v1/recap?kind=&subject_id=    List recaps (paginated, newest first)
PATCH /v1/recap/:id                 User edits headline/bullets/next_actions
DELETE /v1/recap/:id                Forget a recap

POST /v1/resume                     Begin a resume from a recap or raw subject_id
GET  /v1/resume/:handle             Poll resume readiness (loaded transcript, model warm, etc.)
```

**Why a generic `/v1/recap` instead of `/sessions/:id/recap` etc.?** One handler, one schema, one auth path, one client implementation per platform. The `kind` discriminator keeps the table small and the clients dumb.

The watch surface gets a slimmed parallel under `/watch/recap/*` that returns the same shape but with `bullets` truncated to 3 and `artifacts` omitted (watches don't render file lists). Mobile uses the full `/v1/recap` surface.

See per-scope docs for the exact request/response bodies.

## Storage

| Data | Where | Encryption | Why |
|---|---|---|---|
| Session recaps | `~/.vibecli/sessions.db` — new `recaps` table joined on `sessions.id` | None (matches existing `messages` table) | Recaps reference message IDs; co-locate. File-permission protected, like the rest of `sessions.db`. |
| Job recaps | `~/.vibecli/jobs.db` — new `recaps` table | ChaCha20-Poly1305 (matches existing `jobs` table) | Inherits the existing job-store encryption posture. |
| Diff-chain recaps | `<workspace>/.vibecli/workspace.db` — new `diff_chains` + `diff_chain_recaps` tables | ChaCha20-Poly1305 (matches `workspace_secrets`) | Diff chains belong to a workspace; co-locate. |

**No new key-derivation schemes.** Each store reuses its existing `derive_key()` (machine-bound for profile/jobs; machine + workspace-bound for workspace). See `vibecli/vibecli-cli/src/profile_store.rs`, `workspace_store.rs`.

**Migration pattern** — idempotent `ALTER TABLE … ADD COLUMN` and `CREATE TABLE IF NOT EXISTS`, exactly the pattern used in `session_store.rs:231` (`maybe_add_column`) and `session_store.rs:210` (`backfill_fts_if_needed`). No destructive migrations.

## Security & privacy rules

1. **Recaps inherit the sensitivity of their source.** A recap of a chat that touched secrets is itself sensitive and lives in the encrypted store for that scope.
2. **Sharing requires explicit user action.** The existing `/share/:id` route (read-only HTML view of a session) extends to recaps — but only when the user clicks "Share". Default state: never leaves the device.
3. **LLM-generated recaps are routed to the user's currently selected provider** — same provider used for the chat itself. No silent fan-out to a "summarization model".
4. **Recap generation is logged in `messages` as a system event** with role `system` and a `recap.generated` marker — auditable, never hidden.
5. **No recap of recaps.** Recapping a recap is forbidden to prevent information loss spirals.

## Patent-distance posture

Diffcomplete recap/resume (`03-diffcomplete.md`) must remain distant from all five elements of the inline-suggestion patent claim, per the project's standing constraint:

- **Element 1 (continuous monitoring):** Recap is generated only on explicit user trigger or chain end. No background scanning.
- **Element 2 (predictive completion):** Recap summarizes *prior* user-directed diffs; it does not predict future code.
- **Element 3 (inline UI):** Recap renders in a side panel / modal footer / dedicated view — never inlined into the editor buffer.
- **Element 4 (acceptance UI):** Recap has no accept/reject affordances on code; it surfaces *what already happened*, not what *might*.
- **Element 5 (model context window):** Recap is itself a deliberate context-trimming primitive — it shrinks context, doesn't extend it.

Cross-reference: `notes/PATENT_AUDIT_INLINE.md` (gitignored). Any change to the diffcomplete recap surface is required to walk the five-element checklist before merging.

## Sequencing

The three scopes ship independently but share a common foundation. Recommended order:

| Phase | Slice | Touches | Why first |
|---|---|---|---|
| **F1** | Foundation: `recap` + `resume` types, daemon `recaps` table on `sessions.db`, `/v1/recap` routes (Session kind only) | vibecli daemon, vibecli REPL `/recap` command | Everything else depends on the schema and routes; ship them under one `kind` first. |
| **F2** | vibeui Session recap panel + auto-recap toggle in Settings | vibeui only | Validates the daemon contract from a real client before mobile/watch work. |
| **J1** | Job kind: extend `recap` to jobs, table on `jobs.db`, BackgroundJobs panel "Recap" button | daemon + vibeui | Reuses the foundation; no new shape. |
| **D1** | Diffcomplete kind: persisted chains in `workspace.db`, recap of chain at modal close | daemon + vibeui | Patent-distance review required before merge. |
| **M1** | vibemobile: ChatScreen recap header + resume from recap | vibemobile | One surface, narrow scope. |
| **W1** | watchOS + Wear: recap-only view, resume opens phone/desktop session | both watch apps | Watches read but never compose a full recap. |
| **F3** | Auto-resume-last-session toggle, cross-device handoff via `/mobile/sessions` | all clients | Last; depends on every client knowing the recap shape. |

## Per-platform UX summary

Detail lives in the per-scope docs; here's the cross-cutting picture:

| Surface | Recap consumer | Recap producer | Resume initiator |
|---|---|---|---|
| **vibecli REPL** | `/recap [session]` command, end-of-session auto-print | yes (Heuristic + LLM) | `/resume <id>` (existing) + new `--from-recap <id>` |
| **vibecli TUI** | recap pane on session-end screen | yes | "r" hotkey on session list |
| **vibeui chat tab** | recap card at top of restored tab | yes | "Resume" button on history list |
| **vibeui BackgroundJobs** | recap row replaces "Done" badge after completion | yes (auto on terminal status) | "Re-run from recap" button |
| **vibeui DiffComplete** | recap footer in modal after N regenerations | yes (heuristic) | "Resume chain" from chain history |
| **vibemobile ChatScreen** | recap card above transcript | no (read-only) | tap recap → opens chat at resume hint |
| **vibewatch / Wear** | dedicated RecapView (3 bullets max) | no | tap "Continue on phone" → handoff via `/mobile/sessions` |

## Open questions

1. **Auto-recap trigger threshold.** When does a chat session "end"? Idle timeout? Explicit `/end`? Tab close? Different defaults per surface, configurable. → Settled per-doc.
2. **LLM vs heuristic default.** Heuristic recaps are free and offline; LLM recaps are richer but cost tokens. Default = heuristic, user opt-in to LLM. → Confirmed in `01-session.md`.
3. **Counsel sessions.** They're in-memory only today (`counsel.rs`). Persisting them is a prerequisite for Counsel recaps. Out of scope for the first cut; mark as a follow-on.
4. **Cross-workspace recap visibility.** A session may span workspaces (rare). Default: scope to the workspace the session started in. → Settled in `01-session.md`.
5. **Patent posture for diffcomplete-resume.** Re-running a chain from index N — does that violate any element? Audit needed before D1 merges.
