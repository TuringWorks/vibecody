# Project Management and Agile Practices

When managing projects using Agile methodologies:

1. Run Scrum ceremonies with discipline: daily standups (15 min, blockers-focused), sprint planning (size backlog to team capacity), sprint review (demo working software to stakeholders), and retrospectives (actionable improvements, not venting sessions).
2. Apply Kanban WIP limits strictly to prevent context switching and expose bottlenecks; track flow metrics (cycle time, throughput, lead time) to identify systemic issues rather than blaming individuals.
3. Write user stories following the INVEST criteria (Independent, Negotiable, Valuable, Estimable, Small, Testable) using the format "As a [persona], I want [goal] so that [benefit]" with clear acceptance criteria.
4. Estimate work using relative sizing (story points or t-shirt sizes) rather than hours; use planning poker or async estimation to leverage collective wisdom and surface misunderstandings early.
5. Track velocity as a planning tool not a performance metric; use rolling averages over 3-5 sprints for capacity planning and never compare velocity across teams.
6. Groom the backlog regularly by refining the top items to be sprint-ready, splitting epics into stories, removing stale items, and maintaining a prioritized order using MoSCoW (Must/Should/Could/Won't) or RICE (Reach, Impact, Confidence, Effort) scoring.
7. Run retrospectives with varied formats to prevent staleness: Start-Stop-Continue, 4Ls (Liked, Learned, Lacked, Longed for), sailboat (wind/anchors), or timeline; always leave with 2-3 concrete action items with owners.
8. Manage risks and dependencies proactively by maintaining a risk register, flagging cross-team dependencies in planning, building buffer for unknowns, and escalating blockers within 24 hours.
9. Report to stakeholders using consistent formats: sprint goals and progress, key metrics (velocity, burndown), risks and mitigations, and upcoming milestones; tailor detail level to the audience.
10. Define "Definition of Done" explicitly including code review, tests passing, documentation updated, accessibility checked, and product owner acceptance; enforce it consistently to prevent technical debt accumulation.
11. Use burndown charts for sprint tracking and burnup charts for release tracking; identify scope creep early by monitoring the gap between planned and actual work added mid-sprint.
12. Scale Agile thoughtfully using frameworks like SAFe (for large enterprises needing alignment) or LeSS (for simpler multi-team coordination); avoid over-engineering process and always optimize for delivery speed and team autonomy.
