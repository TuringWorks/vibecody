# GitOps with ArgoCD and Flux

When implementing GitOps workflows for Kubernetes deployments:

1. **App-of-Apps Pattern** — Organize ArgoCD deployments hierarchically: a root Application that manages child Applications, each representing a service or infrastructure component. Store the root app manifest in a bootstrap repository, use Helm or Kustomize to template child app definitions, and enable cascading deletes so removing a child from Git removes it from the cluster.
2. **ApplicationSets for Scale** — Use ArgoCD ApplicationSets to generate Applications dynamically from Git directory structures, cluster lists, or pull requests. The Git generator creates an Application per directory in a monorepo, the cluster generator deploys across multiple clusters, and the pull request generator creates preview environments automatically.
3. **Kustomize Overlays** — Structure manifests with a `base/` directory containing shared resources and `overlays/` directories per environment (dev, staging, prod). Use strategic merge patches for environment-specific changes (replicas, resource limits, config), use components for optional features, and avoid variable substitution in favor of patches for clarity.
4. **Helm with GitOps** — Store Helm value files in Git (not the charts themselves) and let ArgoCD/Flux render charts during sync. Pin chart versions in Application specs, use multiple values files layered by environment, and test chart rendering locally with `helm template` before committing. Never use `helm install` directly; the GitOps controller is the single source of truth.
5. **Image Automation** — Use Flux's Image Automation Controller or ArgoCD Image Updater to automatically detect new container images and update manifests in Git. Configure image policies to filter by semver, regex, or timestamp. Commit image updates to a dedicated branch or directory to separate automated changes from human changes for cleaner audit trails.
6. **Progressive Delivery with Argo Rollouts** — Replace Deployments with Rollouts to gain canary and blue-green deployment strategies. Define analysis templates that query Prometheus metrics during rollout steps, set success thresholds (e.g., error rate below 1%), and configure automatic rollback on failure. Integrate with the service mesh or ingress controller for traffic splitting.
7. **Multi-Cluster GitOps** — Manage multiple clusters from a single GitOps control plane. Register clusters in ArgoCD, use ApplicationSets with cluster generators to deploy consistently, and store cluster-specific overrides in labeled directories. For Flux, use multi-tenancy with separate Kustomizations per cluster, and sync cluster credentials securely via Sealed Secrets or SOPS.
8. **RBAC and Multi-Tenancy** — Configure ArgoCD RBAC policies to restrict teams to their namespaces and repositories. Use Projects to scope which repos, clusters, and namespaces a team can deploy to. Map SSO groups to ArgoCD roles, enable resource-level access control, and audit all sync operations through ArgoCD's built-in event logging.
9. **Notification and Alerting** — Deploy ArgoCD Notifications Controller or Flux notification-controller to send sync status updates to Slack, Teams, email, or webhooks. Configure triggers for sync failures, health degradation, and out-of-sync detection. Include diff summaries in notifications so reviewers can assess changes without checking the UI.
10. **Sync Policies and Windows** — Configure automated sync with self-heal (revert manual cluster changes) and prune (delete resources removed from Git). Use sync windows to restrict production deployments to maintenance periods. Set retry policies with backoff for transient failures, and require manual sync approval for critical namespaces via sync waves.
11. **Secret Management in GitOps** — Never commit plaintext secrets to Git. Use Sealed Secrets (encrypt secrets with a cluster-specific key), SOPS with age/PGP (encrypt values in-place), or External Secrets Operator (reference secrets from Vault/AWS SM). Commit the encrypted or reference manifests to Git and let the controller decrypt or fetch at sync time.
12. **Drift Detection and Reconciliation** — Configure reconciliation intervals (Flux: 1-5 minutes, ArgoCD: 3 minutes default) to detect and correct drift between Git and cluster state. Enable ArgoCD's self-heal to automatically revert manual kubectl changes, log all drift events for compliance, and set up alerts when drift is detected more than a configurable number of times per hour.
