# Web Accessibility Best Practices

When building accessible web applications:

1. **Use Semantic HTML as Your Foundation** — Choose the correct HTML element for each purpose: `<button>` for actions, `<a>` for navigation, `<nav>` for navigation regions, `<main>` for primary content, `<article>` for self-contained content, `<aside>` for tangential content, `<header>`/`<footer>` for landmark regions. Semantic elements provide built-in keyboard behavior, screen reader announcements, and browser features for free. Never use `<div onclick>` when `<button>` is the correct choice.
2. **Meet WCAG 2.1 AA Color Contrast Ratios** — Normal text (under 18pt or 14pt bold) requires a minimum contrast ratio of 4.5:1 against its background. Large text (18pt+ or 14pt+ bold) requires 3:1. UI components and graphical objects require 3:1 against adjacent colors. Use tools like the WebAIM Contrast Checker or browser DevTools to verify. For AAA compliance, target 7:1 for normal text and 4.5:1 for large text. Never convey information through color alone; pair with icons, patterns, or text labels.
3. **Implement Complete Keyboard Navigation** — Every interactive element must be reachable and operable via keyboard. Use `Tab`/`Shift+Tab` for sequential navigation, `Enter`/`Space` for activation, `Escape` to dismiss overlays, and arrow keys for navigation within composite widgets (menus, tabs, grids). Ensure `tabindex="0"` for custom interactive elements, `tabindex="-1"` for programmatically focusable elements, and never use `tabindex` values greater than 0. Test by unplugging your mouse and navigating the entire application.
4. **Manage Focus Deliberately** — When content changes dynamically (modals, drawers, route transitions), move focus to the new content. Trap focus inside modal dialogs so Tab does not escape behind the overlay. Restore focus to the trigger element when a modal closes. On single-page app route changes, move focus to the main content heading or use an `aria-live` region to announce the new page. Never let focus get lost in a void after DOM updates.
5. **Apply ARIA Roles, States, and Properties Correctly** — Use ARIA only when native HTML cannot express the semantics: `role="tablist"`, `role="dialog"`, `aria-expanded`, `aria-selected`, `aria-controls`, `aria-describedby`. Follow the first rule of ARIA: if you can use a native HTML element with built-in semantics, do that instead. Always keep ARIA states in sync with visual state (e.g., `aria-expanded="true"` when a dropdown is open). Invalid ARIA is worse than no ARIA because it actively misleads assistive technology.
6. **Write Meaningful Alt Text for Images** — Informative images need alt text describing their content and purpose, not their appearance ("Bar chart showing 40% growth in Q3" not "chart image"). Decorative images need `alt=""` to be hidden from screen readers. Complex images (charts, diagrams) need both concise alt text and a longer description via `aria-describedby` or a visible caption. Icon buttons need alt text describing the action ("Close dialog" not "X icon").
7. **Implement Skip Links and Landmark Navigation** — Add a "Skip to main content" link as the first focusable element on every page, visible on focus. Use landmark roles (`<nav>`, `<main>`, `<aside>`, `<footer>`) so screen reader users can jump between page regions. Label multiple navigation regions with `aria-label` (e.g., `<nav aria-label="Primary">`, `<nav aria-label="Breadcrumb">`). Screen reader users navigate by landmarks as frequently as sighted users scan visual layout.
8. **Make Forms Fully Accessible** — Associate every input with a visible `<label>` using `for`/`id` pairing. Group related fields with `<fieldset>` and `<legend>`. Mark required fields with `aria-required="true"` and visible indicators (not just color). Display validation errors inline next to the field, linked via `aria-describedby`, and announce them with `aria-live="polite"` or by moving focus to the first error. Use `autocomplete` attributes for common fields (name, email, address) to assist autofill.
9. **Support Reduced Motion Preferences** — Wrap all animations and transitions in a `prefers-reduced-motion` media query. When the user prefers reduced motion, disable parallax, auto-playing carousels, animated transitions, and decorative motion. Essential motion (progress indicators, loading spinners) can remain but should be simplified. In CSS: `@media (prefers-reduced-motion: reduce) { * { animation-duration: 0.01ms !important; transition-duration: 0.01ms !important; } }`. In JavaScript, check `window.matchMedia('(prefers-reduced-motion: reduce)').matches`.
10. **Use Live Regions for Dynamic Content** — When content updates without a page reload (notifications, chat messages, form validation, real-time data), use `aria-live` regions to announce changes to screen readers. Use `aria-live="polite"` for non-urgent updates (success messages, status changes) and `aria-live="assertive"` for urgent alerts (errors, time-sensitive warnings). Keep announcements concise and avoid flooding the screen reader output queue with frequent updates.
11. **Test with Automated and Manual Tools** — Run automated accessibility scans in CI with axe-core (catches about 30-40% of issues): `npx @axe-core/cli URL` or integrate `jest-axe` in component tests. Complement with manual testing: keyboard-only navigation, screen reader testing (VoiceOver on macOS, NVDA on Windows, Orca on Linux), browser zoom to 200%, and Lighthouse accessibility audits. Automated tools cannot catch focus management, reading order, or cognitive accessibility issues.
12. **Handle Dynamic Content and SPAs** — In single-page applications, manage document title updates on route changes (`document.title = "Page Name - App Name"`), announce route changes to screen readers (via `aria-live` or focus management), and ensure client-side routing does not break the browser back button or screen reader virtual buffer. Test that lazy-loaded content is accessible after it appears and that loading states are announced.
13. **Design Accessible Data Tables** — Use proper `<table>`, `<thead>`, `<tbody>`, `<th>`, and `<td>` elements for tabular data. Add `scope="col"` or `scope="row"` to header cells. For complex tables, use `id` and `headers` attributes to explicitly link data cells to their headers. Provide a `<caption>` summarizing the table's purpose. For sortable columns, indicate the current sort state with `aria-sort="ascending"` or `aria-sort="descending"`. Never use tables for layout.
14. **Ensure Touch Target Accessibility** — Interactive elements should have a minimum touch target size of 44x44 CSS pixels (WCAG 2.5.5 AAA) or at minimum 24x24 pixels (WCAG 2.5.8 AA). Add padding to small interactive elements to increase the tap area without changing visual size. Ensure adequate spacing between adjacent targets to prevent accidental activation. This benefits users with motor impairments, elderly users, and everyone on mobile devices.
