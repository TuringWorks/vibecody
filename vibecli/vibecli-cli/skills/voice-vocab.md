# Codebase-Specific Voice Recognition

When configuring speech recognition for coding and codebase-specific vocabulary:

1. **Vocabulary Extraction from Symbols** — Scan the codebase to extract a custom vocabulary: function names, class names, module names, variable names (de-duplicated, frequency-sorted). Include project-specific acronyms from README and comments. Export as a plain-text word list compatible with Whisper's `initial_prompt` or the hot-words API of your STT backend. Refresh the vocabulary list whenever the codebase changes significantly (>100 new symbols).
2. **Hot-Words Injection** — Inject high-frequency project symbols as hot-words with boosted probability weights in the STT model. For Whisper, pass the top 200 symbols as the `initial_prompt` string (space-separated, no punctuation). For cloud STT APIs (Google, Azure), use the custom model adaptation or phrase hints API with a boost score of 15–20 for project-specific terms. Monitor for over-boosting: symbols that match too broadly will degrade non-code dictation quality.
3. **Phonetic Normalization of camelCase and snake_case** — Before adding symbols to the vocabulary, generate phonetic variants: split `getUserById` into "get user by id", split `user_profile_service` into "user profile service". Register both the canonical symbol and the phonetic expansion. During post-processing, reverse the mapping: when the STT returns "get user by id", attempt to resolve it back to the matching symbol using fuzzy match against the known symbol list.
4. **Frequency-Based Symbol Weighting** — Weight vocabulary entries by codebase frequency: symbols appearing in >10 files get maximum boost; symbols appearing once get no boost. Recompute weights on each vocabulary refresh. High-frequency symbols should appear earlier in the `initial_prompt` string, as Whisper gives more weight to early context. Do not boost symbols with fewer than 3 characters — they cause excessive false positives.
5. **WER Measurement** — Measure Word Error Rate (WER) periodically using a ground-truth test set of 50–100 representative voice commands recorded from real developer sessions. WER = (substitutions + insertions + deletions) / total reference words. Baseline WER without codebase vocabulary should be measured first. Acceptable WER for coding voice commands: <8% for common commands, <15% for long identifiers. Log WER trends over time; a rising WER signals vocabulary drift.
6. **Ambiguity Resolution for Homophone Symbols** — When two symbols sound identical or similar (e.g., `getUser` and `getUsers`, `foo` and `fuu`), insert a disambiguation step: prompt the user to confirm via UI highlight or a quick follow-up question ("Did you mean `getUser` or `getUsers`?"). Do not silently choose one; wrong symbol insertion in code is a high-cost error. Log all disambiguation events to improve future weighting.
7. **Language and Locale Handling** — Select the Whisper model language explicitly (do not rely on auto-detect for coding sessions). For multilingual teams, maintain per-user language settings and separate vocabulary files per locale. Code symbol pronunciation varies by locale (e.g., `data` vs `dayta`); collect pronunciation variants from team members and add both to the vocabulary. Never mix language auto-detection with hot-word injection — they conflict.
8. **Latency Targets and Streaming** — For interactive coding use, voice recognition latency must be under 300ms end-to-end (audio capture → STT → symbol resolution → editor insertion). Use Whisper's streaming mode or a local small model (Whisper tiny/base) for low-latency paths. Fall back to cloud STT for higher accuracy on complex phrases. Display a visual waveform indicator while recording to reduce user uncertainty about whether the system is listening.
