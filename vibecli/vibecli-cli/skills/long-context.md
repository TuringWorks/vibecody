# Long Context (2M–10M Token) Strategies

When working with extremely large codebases or documents requiring multi-million token contexts:

1. **Model Selection by Context Length** — Match the model to the required context window: use Gemini 1.5 Pro/Flash for 1M–2M token windows, Gemini Ultra or specialized long-context models for 2M–10M. For Claude models, cap at 200K tokens per call and use sliding window strategies for larger inputs. Always check the model's effective context length (not just the advertised maximum) — quality degrades as context approaches the limit. Benchmark on a sample before committing to a full ingestion run.
2. **Cost Estimation Before Dispatch** — Calculate token count (using `tiktoken` or the provider's counting API) before submitting a large context request. Estimate cost = tokens × per-token price (input + output). Surface the estimate to the user and require explicit confirmation for requests costing more than a configurable threshold (default: $1.00). Log estimated vs actual cost for every large-context task to calibrate future estimates.
3. **Semantic Chunking at Function Boundaries** — When the full context exceeds the model window, chunk at semantic boundaries rather than fixed byte offsets: split at function/class/module boundaries using AST parsing. Each chunk must be self-contained: include the relevant imports, type definitions, and surrounding context necessary to understand the chunk. Avoid splitting a function body across two chunks — always include the complete function.
4. **Overlap Strategy** — When using sliding-window chunking, include a 5–10% overlap between adjacent chunks (measured in tokens). The overlap should contain complete logical units (full function signatures, complete docblocks) rather than arbitrary token counts. Overlaps allow the model to maintain cross-chunk coherence. Track which content was included in overlap to avoid double-counting in result aggregation.
5. **Monorepo Ingestion Order** — When ingesting a full monorepo, process files in this priority order: (1) entry points and main modules, (2) shared libraries and interfaces, (3) test files, (4) configuration and build files, (5) documentation. This order maximizes the value of early context tokens when the model must truncate. Generate a dependency graph first and ingest in topological order so that callers are ingested after their dependencies.
6. **Relevance Filtering Before Ingestion** — Before building a large context, run a fast relevance filter (BM25, TF-IDF, or embedding similarity) to exclude files with near-zero relevance to the task query. Target 30–50% context reduction through filtering before chunking. Never ingest auto-generated files (lockfiles >1MB, minified JS, compiled binaries, migration history dumps) — they consume tokens without adding semantic value.
7. **Progressive Disclosure Pattern** — For interactive long-context tasks, start with a compressed representation (file tree + function signatures) and expand into full file content only for files the model requests via tool call. This reduces initial token cost by 70–90% for tasks where the model only needs a subset of the codebase. Track which expansions were requested to identify what a model actually uses vs what was ingested speculatively.
8. **Result Aggregation Across Chunks** — When querying a long context in chunks, aggregate results using a merge step: collect per-chunk answers, run a final synthesis prompt over the collected answers (not re-ingesting the full context), and produce the final output. Track confidence per chunk answer; weight low-confidence chunk results less in the synthesis. For factual queries (e.g., "does this function exist?"), use OR-aggregation; for analytical queries (e.g., "summarize all usages"), use concatenation-then-synthesis.
