# Observability: Structured Logging

When implementing structured logging and log aggregation:

1. **Emit Structured JSON Logs** — Always log in JSON format with consistent top-level fields: `timestamp` (ISO 8601 with timezone), `level`, `message`, `service`, `version`, and `environment`. Use a logging library that supports structured output natively (e.g., `tracing` + `tracing-subscriber` in Rust, `pino` in Node.js, `structlog` in Python, `slog` or `zap` in Go). Never rely on regex parsing of unstructured text in production.
2. **Define a Log Levels Strategy** — Use levels consistently across all services: `TRACE` for fine-grained debugging (never in production), `DEBUG` for development diagnostics, `INFO` for business-meaningful events (request served, job completed), `WARN` for recoverable anomalies (retry succeeded, deprecated API used), `ERROR` for failures requiring attention, `FATAL` for unrecoverable conditions. Default production level to `INFO` and allow runtime adjustment via config or environment variable without redeployment.
3. **Propagate Correlation IDs** — Generate a unique `correlation_id` (UUID v4 or ULID) at the system boundary (API gateway, message consumer) and propagate it through every downstream call via HTTP headers (`X-Correlation-ID`), message metadata, or thread-local/async-local context. Include it in every log line so you can reconstruct the full request path with a single query.
4. **Integrate with Centralized Platforms** — Ship logs to a centralized system (ELK/OpenSearch, Grafana Loki, Datadog, Splunk). Use lightweight shippers (Fluent Bit over Fluentd for edge, Promtail for Loki, Vector for multi-destination routing). In Kubernetes, prefer a DaemonSet log collector reading from stdout/stderr rather than sidecar containers to reduce resource overhead.
5. **Configure Log Rotation and Retention** — When writing to disk, configure rotation by size (100-500 MB) and time (daily), keeping 7-14 days locally. In centralized systems, define retention tiers: hot storage (7-30 days, fast queries), warm storage (30-90 days, slower queries), cold/archive (1+ years for compliance). Automate purging to control storage costs.
6. **Redact PII and Sensitive Data** — Implement a log sanitizer that automatically redacts or masks personally identifiable information (emails, SSNs, credit card numbers, auth tokens) before logs leave the process. Use allowlist-based field logging rather than blocklist-based redaction. Audit log output quarterly for accidental PII leakage. Never log request/response bodies in production without explicit redaction.
7. **Implement Log Sampling for High-Volume Paths** — For high-throughput endpoints (health checks, metrics scrapes, static assets), sample logs at 1-10% to avoid flooding your aggregation pipeline. Always log 100% of errors and warnings regardless of sampling. Use deterministic sampling based on the correlation ID so all logs for a sampled request are either included or excluded together.
8. **Propagate Context Across Async Boundaries** — In async runtimes, ensure logging context (correlation ID, user ID, tenant ID) survives across `await` points, thread pool handoffs, and message queue boundaries. Use context-aware logging (e.g., `tracing::Span` in Rust, `AsyncLocalStorage` in Node.js, `contextvars` in Python). Verify context propagation in integration tests.
9. **Set Up Log-Based Alerting** — Configure alerts on log patterns that indicate problems: error rate spikes (>5x baseline over 5 minutes), specific fatal messages, security-sensitive events (auth failures, permission denied bursts), and absence of expected periodic logs (heartbeat monitoring). Use deduplication and cooldown windows to prevent alert storms.
10. **Minimize Performance Impact** — Log asynchronously using a buffered writer with a bounded queue (drop oldest on overflow rather than blocking the application thread). Avoid string formatting for disabled log levels (use lazy evaluation or macros). Benchmark logging overhead; aim for <1% CPU impact under normal load. Disable verbose field serialization (stack traces, full payloads) unless the level is DEBUG or lower.
11. **Standardize Log Schemas Across Services** — Publish a shared log schema (JSON Schema or protobuf) that all services must conform to. Include mandatory fields (`timestamp`, `level`, `service`, `correlation_id`) and optional domain-specific fields. Validate log output in CI with a schema linter. A consistent schema enables cross-service queries without per-service parsing rules.
12. **Add Request/Response Metadata to Logs** — For HTTP services, log request metadata on every request: method, path, status code, duration_ms, request_size, response_size, client_ip (hashed if PII-sensitive), and user_agent. This single log line per request powers latency dashboards, error rate tracking, and traffic analysis without needing a separate metrics pipeline.
13. **Use Contextual Log Enrichment** — Enrich logs at the infrastructure level with metadata the application does not need to know about: Kubernetes pod name, node, namespace, container ID, deployment version, cloud region. Use Fluent Bit or Vector filters to inject these fields from environment variables or Kubernetes downward API, keeping application code clean.
14. **Test Your Logging Pipeline End-to-End** — Include logging pipeline tests in your integration test suite: emit a log with a unique marker, query the aggregation system after a short delay, and assert the log arrived with all expected fields intact. This catches serialization bugs, shipper misconfigurations, and index mapping issues before they cause blind spots in production.
