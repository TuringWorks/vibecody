# SOX, PCI DSS, and SOC 2 Compliance

When implementing regulatory compliance controls for SOX, PCI DSS, and SOC 2:

1. **Implement SOX IT General Controls (ITGCs)** — SOX compliance for IT requires controls in four domains: access management (role-based access, quarterly access reviews, privileged account monitoring), change management (documented change requests, testing, approval before production deployment), computer operations (job scheduling, backup monitoring, incident management), and program development (SDLC documentation, security testing, code review). Map each control to a specific policy, procedure, and evidence artifact. Automate evidence collection to reduce audit preparation from weeks to hours.
2. **Meet PCI DSS Network Segmentation Requirements** — Isolate the Cardholder Data Environment (CDE) from all other networks using firewalls, VLANs, or cloud VPC security groups. Document all network flows into and out of the CDE. Restrict access to the CDE to only the systems and personnel that require it. Conduct penetration testing of segmentation controls every 6 months (PCI DSS 4.0 requirement). Use micro-segmentation for fine-grained control. Maintain a current network diagram that shows all connections to the CDE and review it quarterly.
3. **Encrypt Cardholder Data Per PCI DSS Requirements** — Encrypt stored cardholder data (PAN) using strong cryptography (AES-256) with documented key management procedures: key generation with sufficient entropy, secure key storage (HSM or cloud KMS), key rotation at least annually, split knowledge and dual control for key management operations, and cryptographic key destruction procedures. Encrypt PAN in transit using TLS 1.2+ with strong cipher suites. Never store the full magnetic stripe, CVV, or PIN data after authorization. Mask PAN when displayed (show only first six and last four digits).
4. **Design SOC 2 Trust Services Criteria Controls** — Map your controls to the five SOC 2 trust criteria: Security (access control, vulnerability management, encryption), Availability (uptime monitoring, disaster recovery, capacity planning), Processing Integrity (input validation, error handling, reconciliation), Confidentiality (data classification, encryption, access restrictions), and Privacy (consent management, data retention, subject access). Document each control's design, implementation, operating effectiveness, and monitoring. Maintain a controls matrix with owner, frequency, and evidence location.
5. **Build Immutable Audit Trails** — Log all security-relevant events to a tamper-evident audit trail: authentication attempts (success and failure), authorization decisions, data access and modifications, configuration changes, privileged operations, and system events. Write audit logs to append-only storage (S3 with Object Lock, immutable Elasticsearch indexes, or blockchain-anchored logs). Include timestamp, actor, action, resource, outcome, and source IP in every entry. Retain audit logs for the required period (SOX: 7 years, PCI: 1 year, SOC 2: per policy).
6. **Enforce Separation of Duties in CI/CD** — Implement controls that prevent a single person from writing, approving, and deploying code to production. Require pull request approval from a reviewer who is not the author. Separate deployment permissions from code commit permissions. Use branch protection rules to enforce review requirements. Log the full chain of custody: who committed, who approved, who deployed, and when. For SOX-critical applications, require manager approval for production deployments. Audit separation of duties controls quarterly.
7. **Automate Compliance Checks in CI/CD** — Integrate compliance validation into your deployment pipeline: static analysis for security vulnerabilities (SAST), dependency scanning for known CVEs, infrastructure-as-code scanning (Checkov, tfsec, OPA) for misconfigurations, container image scanning for vulnerabilities, secrets detection in code (git-secrets, truffleHog), and license compliance checking. Fail the pipeline on critical findings. Generate compliance reports as build artifacts. Use policy-as-code (Open Policy Agent, Kyverno) to enforce organizational standards automatically.
8. **Streamline Evidence Collection** — Replace manual evidence gathering with automated collection: pull access review lists from IAM systems via API, export change management records from CI/CD platforms, collect vulnerability scan results from security tools, and aggregate monitoring configurations from observability platforms. Store evidence in a structured repository (GRC platform or organized file storage) indexed by control ID and audit period. Automate evidence freshness checks to flag stale or missing artifacts before auditor requests.
9. **Conduct Access Control Reviews** — Perform quarterly access reviews for all systems in scope: verify that user access aligns with current job responsibilities, remove terminated employee access within 24 hours (automate via HR system integration), review and certify privileged access with manager approval, rotate service account credentials on a defined schedule, and document the review process and outcomes. For PCI DSS, review CDE access specifically. For SOX, focus on financial application access. Use identity governance platforms (SailPoint, Okta Identity Governance) to automate the review workflow.
10. **Implement Vulnerability Management Programs** — Establish a vulnerability management lifecycle: discovery (weekly automated scans with Nessus, Qualys, or cloud-native tools), prioritization (CVSS score adjusted for exploitability and asset criticality), remediation (critical: 48 hours, high: 7 days, medium: 30 days, low: 90 days), verification (rescan after fix), and reporting (monthly executive summary, quarterly trend analysis). For PCI DSS, run internal and external vulnerability scans quarterly and after significant changes. Maintain an exception process for vulnerabilities that cannot be remediated within SLA.
11. **Meet Penetration Testing Requirements** — Conduct penetration testing at least annually (PCI DSS requires both internal and external tests) and after significant infrastructure or application changes. Use qualified testers (PCI DSS requires ASV for external scans). Scope tests to cover: network layer (segmentation, firewall rules), application layer (OWASP Top 10), social engineering (phishing simulations), and physical security where applicable. Track all findings to remediation. Retest critical and high findings within 30 days of remediation. Maintain penetration test reports for the audit retention period.
12. **Implement Compliance-as-Code** — Codify compliance requirements as executable policies: use Open Policy Agent (OPA) for infrastructure and API policy enforcement, Sentinel for Terraform policy, Kyverno for Kubernetes admission control, and custom CI checks for application-level policies. Version control all policies alongside application code. Test policy changes in staging before production. Generate compliance posture dashboards from policy evaluation results. This approach scales compliance across teams without manual review bottlenecks.
13. **Manage Change Control for Regulated Systems** — For SOX and PCI in-scope systems, implement a formal change management process: change request with business justification and risk assessment, impact analysis and testing plan, peer review and approval (by a person independent of the requester), scheduled deployment window with rollback plan, post-deployment verification, and change record retention. Classify changes by risk level (standard, normal, emergency) with appropriate approval paths. Track change success rates and failed change causes to improve the process.
14. **Prepare for and Support Audits Efficiently** — Maintain audit readiness year-round rather than scrambling before audits. Keep a living controls matrix with current evidence links. Conduct internal control testing quarterly (self-assessments). Assign a compliance champion in each engineering team responsible for maintaining evidence in their domain. When auditors make requests, provide evidence within 24 hours from your automated collection system. Track auditor findings, remediation actions, and closure dates. Conduct a lessons-learned review after each audit cycle to improve the process.
