# Microsoft Agent Framework (MSAF) 1.0 Compatibility

When integrating agents with Microsoft Agent Framework 1.0:

1. **Manifest Generation** — Every MSAF-compatible agent must publish a JSON manifest at `/.well-known/agent-manifest.json`. Required fields: `agent_id` (UUID), `display_name`, `description`, `version` (semver), `capabilities` (array of capability strings), `input_schema` (JSON Schema), `output_schema`, `health_endpoint` URL, and `auth_methods` (array of supported auth flows). Validate the manifest against the MSAF 1.0 JSON Schema before registration.
2. **Azure AD Token Validation** — Accept only Azure AD-issued JWT tokens (v2.0 endpoint). Validate: `iss` matches the expected tenant ID, `aud` matches your agent's app registration client ID, `exp` is in the future, and the token signature verifies against the JWKS endpoint. Cache JWKS keys with a 1-hour TTL and refresh on signature validation failure. Reject tokens missing required claims with HTTP 401 and a descriptive `WWW-Authenticate` header.
3. **Catalog Registration Patterns** — Register in the Azure agent catalog via the MSAF Management API using the manifest URL plus a registration payload containing: owner principal (service principal or group), environment tag (dev/staging/prod), and discovery tags (free-form key-value pairs for search). After registration, verify the catalog entry is reachable by running a discovery query before marking registration complete. Re-register on manifest version bumps.
4. **Health Endpoint Requirements** — Implement `GET /health` returning HTTP 200 with a JSON body: `{"status": "healthy"|"degraded"|"unhealthy", "version": "...", "uptime_seconds": N, "checks": [...]}`. The `checks` array must include results for all critical dependencies (LLM backend, database, secret store). MSAF catalog health polling occurs every 60 seconds; respond within 2 seconds or the agent is marked degraded. Never return 200 with an `"unhealthy"` status body — use the correct HTTP status code.
5. **Wrapping MCP and A2A in MSAF Envelope** — When exposing MCP tools or A2A tasks through MSAF, wrap each invocation in the MSAF task envelope: `msaf_task_id`, `correlation_id` (for distributed tracing), `caller_identity` (Azure AD OID), `delegated_permissions` (scopes from the token), and the inner `payload` (MCP request or A2A task). The MSAF envelope provides the audit and identity layer; the inner protocol handles task semantics. Never bypass the envelope for performance — it is the compliance boundary.
6. **Capability Advertisement Accuracy** — Only advertise capabilities the agent can currently fulfill. Before registering, run a self-test for each advertised capability. If a dependency is unavailable (e.g., a required tool or model), remove that capability from the manifest and re-register rather than advertising it and failing at runtime. Use MSAF capability deprecation markers (`deprecated: true`) for capabilities being phased out, with a `sunset_date` field.
7. **Multi-Tenant Isolation** — For agents serving multiple Azure AD tenants, enforce strict tenant isolation: each tenant gets a separate session context, separate audit log partition, and separate resource quota. Never allow cross-tenant data access even when the same service principal is used. Validate the `tid` claim in every token and reject tokens whose tenant is not on the pre-registered allow list.
8. **Graceful Version Migration** — When updating to a new manifest version, run old and new versions in parallel for a configurable overlap window (default 48 hours). Direct new registrations to the new version while existing sessions complete on the old version. Emit deprecation warnings to callers still using the old version. Archive old manifests in version control; never delete them, as audit logs reference manifest versions.
