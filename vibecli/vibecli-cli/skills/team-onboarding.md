# AI-Driven Team Onboarding

When designing or executing AI-assisted onboarding experiences for new team members:

1. **New Member Detection Heuristics** — Detect a new member by: first commit date within the last 30 days, no prior commits to the repository (git log --author), workspace creation timestamp less than 2 weeks ago, or an explicit `onboarding: true` flag in the user profile. When detected, activate onboarding mode automatically and surface it to the new member for opt-in confirmation — never force onboarding mode on an experienced developer who changed machines.
2. **Knowledge Gap Identification** — At onboarding start, assess knowledge gaps via: analyzing which files the member has opened vs the hotspot map (files touched most by veterans), comparing recent git activity against required skills for open issues, and a short async questionnaire (5 questions max) about their background in the tech stack. Focus gap analysis on the skills required for the team's current sprint, not the entire codebase — depth over breadth in the first month.
3. **Hotspot-First Learning** — Generate the learning path starting from hotspot files: the 20% of files changed by >80% of commits in the last 90 days. These are the files a new member will encounter immediately in real work. Pair each hotspot file with: a plain-language explanation of its purpose, the key patterns it uses, common pitfalls, and links to relevant past PRs. Learning the hotspots first gives maximum practical return before exploring peripheral areas.
4. **Checkpoint-Based Learning Paths** — Structure the onboarding path as a sequence of checkpoints, each with: a learning objective, 1–3 tasks to complete (read a file, make a small change, run a test), and a self-assessment question. A checkpoint is passed when the new member completes the tasks and answers the assessment correctly. Progress is tracked in the profile store. The path adapts based on checkpoint outcomes — if a member breezes through checkpoints, skip lower-level ones and advance to architecture-level content.
5. **Guide Refresh Cadence** — Refresh onboarding guides quarterly or when: a hotspot file changes significantly (>30% line churn), a new major dependency is added, the build process changes, or the team's primary workflow changes. Tag each guide section with a `last_reviewed` timestamp. Surface stale guide sections (last reviewed >90 days ago) to maintainers for review. Never serve a guide section that has not been reviewed in over 6 months without a staleness warning.
6. **Contextual Help Injection** — As the new member edits files, inject contextual help for the patterns they encounter: when they open a file with a complex pattern (e.g., an actor system, a macro-heavy Rust file, a complex React context), offer a one-paragraph explanation without interrupting their flow. Use non-blocking UI (sidebar hint, status bar message) rather than modal dialogs. Context help is offered once per pattern per session; dismiss permanently when the member indicates they understand it.
7. **Social Graph for Ownership** — Identify code owners for each hotspot file via `git log --follow` attribution and `CODEOWNERS` files. Surface the owner's name and recent PRs to the new member when they are working in that file. This creates natural introduction opportunities: "You're editing `auth.rs` — Alice wrote most of this. Her last PR #234 explains the session model." Human connection accelerates onboarding more than documentation alone.
8. **Onboarding Metrics and Completion** — Track onboarding metrics: time to first commit, time to first PR merged, number of guide sections viewed vs total, checkpoint completion rate, and time-to-productivity (defined as: first week with commit count matching team average). Report these metrics to the member and their manager weekly. Onboarding is complete when the member passes all required checkpoints and their week-over-week commit velocity matches the team average for two consecutive weeks.
