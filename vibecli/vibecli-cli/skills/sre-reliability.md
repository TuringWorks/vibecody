# Site Reliability Engineering (SRE) Practices

When implementing SRE practices and building reliable systems:

1. **Manage SLOs and Error Budgets Rigorously** — Define SLOs for every user-facing service based on user expectations, not engineering aspirations. Track error budget consumption on a 30-day rolling window. When the error budget is nearly exhausted (>80% consumed), freeze feature releases and prioritize reliability work. When error budget is healthy (<50% consumed), teams have license to move fast and take risks. Publish error budget status on a dashboard visible to engineering leadership and product managers. Review SLOs quarterly and adjust based on user feedback and business requirements.
2. **Build Incident Response Playbooks** — Create structured playbooks for every known failure mode: define severity levels (SEV1-SEV4) with clear criteria, escalation paths, communication templates, and resolution procedures. Include diagnostic commands, runbook links, and rollback instructions. Assign roles during incidents: Incident Commander (coordinates), Communications Lead (updates stakeholders), and Operations Lead (executes fixes). Practice incident response with tabletop exercises quarterly. Store playbooks in a searchable, version-controlled system accessible during outages.
3. **Conduct Blameless Postmortems** — After every SEV1/SEV2 incident, write a postmortem within 48 hours covering: timeline of events, root cause analysis (not "human error" — dig deeper to systemic causes), impact assessment (users affected, duration, revenue impact), what went well, what went poorly, and action items with owners and deadlines. Share postmortems broadly across engineering. Track action item completion rates. A blameless culture means focusing on system improvements, not individual fault, while still maintaining accountability for follow-through.
4. **Implement Chaos Engineering** — Proactively inject failures in non-production (and eventually production) environments to validate resilience: kill random pods (Chaos Monkey), inject network latency (Gremlin, LitmusChaos), simulate dependency outages, fill disks, exhaust connection pools, and revoke credentials. Start with game days — planned chaos experiments with the team watching. Define steady-state hypotheses before experiments (e.g., "error rate stays below 1% when database replica fails"). Graduate to automated chaos tests in CI/CD once confidence grows.
5. **Automate Runbooks into Self-Healing Systems** — For every recurring incident type, progress through: (1) document the manual fix in a runbook, (2) script the fix for one-click execution, (3) trigger the script automatically from an alert, (4) build the resilience into the system itself (auto-scaling, circuit breakers, failover). Track how many incidents require human intervention versus self-healing. Target reducing manual incident response by 25% per quarter. Use tools like Rundeck, PagerDuty Automation, or custom operators for automated remediation.
6. **Design On-Call Rotations That Do Not Burn Out Engineers** — Limit on-call shifts to 1 week with at least 2 weeks off between rotations. Target fewer than 2 pages per on-call shift. If page volume is higher, prioritize alert quality improvements and system reliability work. Compensate on-call with time off or additional pay. Provide clear escalation paths so no one is stuck alone. Conduct on-call handoff meetings to share context about ongoing issues. Track page frequency, time-to-acknowledge, and time-to-resolve as team health metrics.
7. **Identify and Reduce Toil Systematically** — Toil is manual, repetitive, automatable work that scales linearly with service growth. Track toil in a toil ledger: for each task, record frequency, duration, automation difficulty, and impact. Target spending less than 50% of SRE time on toil (Google's guideline). Prioritize automating the highest-frequency, lowest-complexity toil first. Common toil sources: manual deployments, certificate rotations, user access provisioning, capacity adjustments, and log investigation. Measure toil reduction over time and celebrate wins.
8. **Plan Capacity Proactively** — Forecast resource needs 3-6 months ahead based on traffic growth trends, planned feature launches, and seasonal patterns. Maintain at least 30% headroom above peak utilization for CPU, memory, and storage. Use load testing to validate capacity limits quarterly. Implement auto-scaling for stateless services with appropriate min/max bounds. For stateful services (databases), plan scaling events with sufficient lead time. Document scaling runbooks and test them before they are needed.
9. **Test Disaster Recovery Regularly** — Document Recovery Time Objective (RTO) and Recovery Point Objective (RPO) for every critical service. Test disaster recovery procedures at least quarterly: database restore from backup (verify data integrity, measure restore time), region failover (DNS cutover, data replication verification), and full environment rebuild from infrastructure-as-code. Track actual recovery times against RTO targets. Automate recovery procedures and test the automation, not just the manual steps.
10. **Implement Progressive Rollouts** — Deploy changes gradually to limit blast radius: canary deployments (1% of traffic, monitor for 15-30 minutes), then staged rollout (10%, 25%, 50%, 100%) with automated rollback on error rate or latency regression. Use feature flags to decouple deployment from release. Define rollback criteria in advance (error rate >2x baseline, latency >3x p99). Automate rollback triggers but allow manual override. Track deployment success rates and mean time to rollback.
11. **Use Feature Flags for Reliability** — Feature flags are not just for product experiments — use them as reliability controls: kill switches to disable non-critical features under load, gradual rollouts that can be reversed without deployment, A/B testing infrastructure changes, and emergency overrides for known failure modes. Implement flag evaluation with local caching and fallback defaults so the flag service itself is not a single point of failure. Clean up stale flags quarterly to prevent flag debt.
12. **Build Observability-Driven Alerting** — Alert on symptoms (error rate, latency, availability) not causes (CPU usage, disk space). Use multi-window burn rate alerting for SLO-based alerts. Every alert must have: a clear description of what is broken, a link to the relevant runbook, and a link to the relevant dashboard. Suppress alerts during maintenance windows. Review alert quality monthly: snooze or delete alerts with >50% false positive rate. An alert that wakes someone up at 3 AM must be worth waking up for.
13. **Implement Dependency Management and Circuit Breaking** — Map all service dependencies (both direct and transitive) and classify them as critical (service cannot function without it) or non-critical (degraded experience is acceptable). Implement circuit breakers on all outbound calls with appropriate thresholds (5 failures in 10 seconds opens the circuit, probe every 30 seconds). Provide fallback behavior for non-critical dependencies (cached data, default values, graceful feature disabling). Test dependency failure scenarios regularly.
14. **Foster a Reliability Culture Across Engineering** — SRE is not just the SRE team's job. Embed reliability practices into all engineering teams: require SLOs before launching new services, include reliability review in design docs, share on-call responsibilities between SRE and development teams, make postmortems a learning opportunity for all engineers, and include reliability metrics in team performance reviews. The goal is shared ownership where every engineer thinks about failure modes and builds resilient systems by default.
