# Distributed Systems Patterns

When designing and operating distributed systems:

1. **Consensus with Raft** — Use Raft for leader election and replicated state machines when you need strong consistency across nodes. Implement heartbeat timeouts (150-300ms typical), log replication with majority quorum, and snapshot-based log compaction. Use established libraries (etcd/raft, hashicorp/raft) rather than implementing from scratch. Prefer Raft over Paxos for its understandability and debuggability.
2. **Distributed Transactions** — Avoid distributed transactions when possible by designing around them (single-writer, saga pattern). When unavoidable, use two-phase commit (2PC) only within a tightly controlled infrastructure, and implement timeout-based abort for the coordinator failure case. Consider deterministic database approaches (Calvin) for high-throughput transactional workloads.
3. **Eventual Consistency Strategies** — Design for eventual consistency by making operations commutative or idempotent. Use version vectors to detect conflicts, implement last-writer-wins for simple cases, and application-level conflict resolution for complex ones. Communicate consistency guarantees to users (e.g., "changes may take up to 5 seconds to appear") and provide read-your-writes consistency where needed via sticky sessions or causal consistency.
4. **Vector Clocks and Causality** — Track causality across distributed nodes using vector clocks or hybrid logical clocks (HLC). Each node maintains a logical timestamp incremented on local events and merged on message receipt. Use vector clocks to detect concurrent writes that need conflict resolution, and prefer HLCs when you also need real-time ordering for human-readable timestamps.
5. **CRDTs for Conflict-Free Replication** — Use Conflict-free Replicated Data Types for state that must be writable on multiple nodes without coordination. Choose G-Counters for increment-only counters, PN-Counters for bidirectional counters, OR-Sets for add/remove sets, and LWW-Registers for last-writer-wins values. CRDTs guarantee convergence but may use more memory than traditional data structures.
6. **Gossip Protocols** — Use gossip (epidemic) protocols for membership detection, failure detection, and metadata dissemination across large clusters. Each node periodically exchanges state with random peers, achieving O(log N) convergence. Implement SWIM-style protocols for failure detection with configurable suspicion timeouts. Gossip works well for 10-10,000 node clusters.
7. **Leader Election** — Implement leader election for coordinating distributed work (single-writer databases, job scheduling, shard rebalancing). Use etcd/ZooKeeper leases with TTLs, implement fencing tokens to prevent split-brain, and ensure followers can detect and respond to leader failures within acceptable time bounds. Always design for the leaderless fallback case.
8. **Circuit Breakers** — Wrap remote calls with circuit breakers that track failure rates and trip open when a threshold is exceeded (e.g., 50% failures in a 10-second window). In the open state, fail fast without making the call. After a cooldown period, enter half-open state and allow a probe request to test recovery. Log all state transitions and alert on trips.
9. **Bulkhead Isolation** — Partition system resources (thread pools, connection pools, memory) per downstream dependency so a failure in one dependency cannot exhaust resources needed for others. Use separate HTTP client instances per service, configure connection pool limits per backend, and implement queue-based load leveling to absorb traffic spikes without cascading overload.
10. **Backpressure Mechanisms** — Propagate load information from slow consumers back to producers using bounded buffers with rejection, reactive streams protocols, HTTP 429 responses with Retry-After headers, or TCP-level flow control. Never use unbounded queues in production. Monitor queue depths and apply adaptive rate limiting that adjusts dynamically based on downstream capacity.
11. **Idempotency Keys** — Make all mutating API operations idempotent by requiring clients to send a unique idempotency key (UUID) with each request. Store the key and response in a server-side cache (Redis with TTL or database table) and return the cached response for duplicate keys. This enables safe retries across network failures without duplicate side effects.
12. **Partition Tolerance Design** — Design for network partitions as an inevitability, not an edge case. Decide per data type whether to favor consistency (reject writes during partition) or availability (accept writes and reconcile later). Implement partition detection via health checks, test partition scenarios using chaos engineering tools (Toxiproxy, tc netem), and document the system's behavior during each failure mode.
