# Platform Engineering

When building Internal Developer Platforms and self-service infrastructure:

1. **Internal Developer Platform Design** — Build an IDP as a curated set of tools, workflows, and self-service capabilities that reduce cognitive load for development teams. Start with the highest-friction developer pain points (environment provisioning, deployment, observability), build thin integration layers over existing tools rather than replacing them, and measure success by deployment frequency and time-to-first-commit for new engineers.
2. **Backstage Developer Portal** — Deploy Backstage as the unified entry point for your IDP. Register all services in the software catalog using `catalog-info.yaml` files, build software templates for golden paths, integrate TechDocs for documentation-as-code, and develop custom plugins for organization-specific workflows. Keep the catalog accurate by automating entity registration from CI/CD pipelines.
3. **Crossplane for Infrastructure** — Use Crossplane to define and provision cloud infrastructure as Kubernetes custom resources. Write Compositions that abstract provider-specific details behind simple claims (e.g., `DatabaseClaim` that provisions RDS or Cloud SQL based on environment). Store XRDs and Compositions in Git, version them, and expose them to developers as self-service APIs.
4. **Golden Paths and Templates** — Design opinionated golden paths that encode best practices into scaffolding templates. A golden path for a new microservice should generate: project skeleton, CI/CD pipeline, Dockerfile, Kubernetes manifests, observability configuration, and catalog registration. Make golden paths the easiest option, not the only option; allow deviations with justification.
5. **Self-Service Infrastructure** — Enable developers to provision databases, caches, queues, and other infrastructure through a self-service interface (CLI, portal, or API) without filing tickets. Implement guardrails (instance size limits, approved regions, cost caps) in the provisioning workflow. Use Terraform modules or Crossplane compositions as the backend, with approval gates only for production resources.
6. **Score-Spec for Workload Definition** — Adopt the Score specification to let developers define workload requirements (containers, resources, service dependencies) in a platform-agnostic format. Translate Score files to platform-specific manifests (Kubernetes, Docker Compose, Helm) using score-compose or score-k8s. This decouples developer intent from infrastructure implementation.
7. **Kratix for Platform APIs** — Use Kratix to build platform-as-a-product by defining Promises: declarative APIs that developers use to request capabilities (databases, CI pipelines, monitoring). Each Promise includes a pipeline that provisions resources and applies organizational policies. Kratix orchestrates across multiple clusters, enabling a federated platform model.
8. **Paved Roads Over Mandates** — Build paved roads that make the right thing the easy thing, rather than enforcing mandates that teams work around. Provide pre-configured CI/CD templates, pre-built Docker base images, standard library packages, and reference architectures. Track adoption metrics; if teams are not using a paved road, investigate why and improve it rather than mandating it.
9. **Platform APIs and Contracts** — Define clear API contracts between the platform team and product teams. Publish platform capabilities as versioned APIs with documentation, SLAs, and deprecation policies. Use Kubernetes CRDs, REST APIs, or CLI tools as the interface. Treat internal platform services with the same rigor as external-facing APIs: versioning, backward compatibility, and change management.
10. **Developer Experience Metrics** — Measure platform effectiveness with DORA metrics (deployment frequency, lead time, change failure rate, MTTR), developer satisfaction surveys (quarterly NPS), onboarding time (time from first commit to production deploy), and self-service adoption rates. Use these metrics to prioritize platform improvements and justify investment to leadership.
11. **SRE Integration** — Embed SRE practices into the platform: provide SLO/SLI templates that teams configure per service, automate error budget tracking, integrate on-call tooling (PagerDuty, OpsGenie) with the service catalog, and provide runbook templates. The platform should make reliability engineering accessible to all teams, not just those with dedicated SREs.
12. **Platform Team Operating Model** — Run the platform team as a product team: maintain a roadmap, gather user feedback, prioritize by impact, and ship iteratively. Staff with a mix of infrastructure engineers, developer experience advocates, and product thinkers. Avoid becoming a ticket queue; instead, build self-service capabilities that eliminate the most common requests and free the team to work on high-leverage improvements.
