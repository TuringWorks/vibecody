# Claw Code Framework Compatibility

When integrating VibeCody agents with the Claw Code open-source agent framework:

1. **JSON-RPC Worker Protocol Registration** — Register as a Claw Code worker by implementing the JSON-RPC 2.0 worker protocol. Required methods: `worker.capabilities` (returns capability manifest), `worker.execute` (runs a task), `worker.cancel` (cancels an in-progress task), and `worker.health` (returns health status). All requests and responses must be valid JSON-RPC 2.0 envelopes with `id`, `method`, and `params`/`result`/`error` fields. Use `null` IDs only for notifications, never for calls expecting a response.
2. **Capability Advertisement Accuracy** — The `worker.capabilities` response must list only capabilities the worker can currently fulfill. Each capability entry must include: `name` (string, e.g., `"code.review"`), `version` (semver), `input_schema` (JSON Schema), `output_schema` (JSON Schema), and `avg_latency_ms` (integer, measured from recent executions). Advertise a capability only after a self-test passes at worker startup. Never advertise capabilities whose dependencies (models, tools, APIs) are unavailable.
3. **Task Type Mapping to Internal Commands** — Map each Claw Code task type to the corresponding internal VibeCody command at registration time. Maintain a static mapping table: `{"code.review" → "aireview", "code.explain" → "explain", "code.refactor" → "refactor", ...}`. If a received task type has no mapping, return a JSON-RPC error with code `-32601` (Method not found) and a message listing supported task types. Do not attempt to auto-infer an unmapped task type.
4. **Bidirectional Client Mode** — When acting as a Claw Code client (consuming tasks from another worker), enable bidirectional mode: the local worker both accepts tasks from Claw Code and dispatches subtasks to other workers via Claw Code routing. In bidirectional mode, implement backpressure: if the local task queue exceeds a configurable threshold (default: 10 pending tasks), return a `worker.busy` notification to the orchestrator and pause accepting new tasks until the queue drains below the threshold.
5. **Error Code Compliance** — Use standard JSON-RPC 2.0 error codes for protocol errors (-32700 to -32600) and Claw Code-specific error codes for task errors (1000–1999). Required Claw Code error codes: `1001` (capability unavailable), `1002` (input validation failed), `1003` (task timeout), `1004` (resource limit exceeded), `1005` (dependency failed). Include a `data` field in every error with: `task_id`, `worker_id`, and a human-readable `detail` string.
6. **Heartbeat and Keep-Alive** — Send a heartbeat notification (`worker.heartbeat`) to the Claw Code orchestrator every 15 seconds when idle and every 5 seconds when executing a task. The heartbeat payload must include: `worker_id`, `status` (idle/busy/draining), `active_tasks` (count), and `uptime_seconds`. If the orchestrator does not receive a heartbeat within 45 seconds, it will mark the worker as unavailable and reroute tasks. Implement heartbeat on a dedicated background thread to ensure it is not blocked by task execution.
7. **Session and Context Propagation** — When receiving a task from Claw Code, extract the `session_id`, `trace_id`, and `parent_task_id` from the request context and propagate them to all downstream tool calls and subtasks. This enables end-to-end distributed tracing across the Claw Code task graph. Never generate a new `trace_id` for a task received from Claw Code — always inherit the one provided. Emit all internal spans with the inherited trace context.
8. **Version Negotiation and Compatibility** — During worker registration, negotiate the Claw Code protocol version via the `worker.capabilities` handshake. Advertise the maximum supported version and accept the orchestrator's selected version. Maintain backward compatibility for at least two major versions of the Claw Code protocol. When the orchestrator selects an older version, disable features not available in that version and log a compatibility warning. Never silently fail when receiving features from an unsupported version — return a JSON-RPC error instead.
