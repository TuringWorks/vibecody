# Backend Performance Optimization

When optimizing backend API and server performance:

1. **Configure Connection Pooling Properly** — Use connection pools for every external resource: databases, Redis, HTTP clients, gRPC channels. Size pools based on expected concurrency and backend capacity, not arbitrary large numbers. A common formula for database pools: `pool_size = (core_count * 2) + disk_spindles`. Set appropriate idle timeouts (30-60 seconds), max lifetime (30 minutes), and connection validation queries. Monitor pool utilization and wait times; if the pool is consistently >80% utilized, investigate slow queries before increasing pool size.
2. **Eliminate N+1 Query Problems** — Detect N+1 patterns where a loop executes one query per item instead of a single batch query. Fix with eager loading (SQL JOINs, ORM includes/preloads), batch loading (DataLoader pattern, `WHERE id IN (...)`), or query consolidation. In ORMs, enable query logging in development and flag endpoints executing more than 10 queries. Use tools like `bullet` (Rails), `nplusone` (Python), or query count assertions in tests.
3. **Design Database Indexes Strategically** — Index columns used in WHERE, JOIN, ORDER BY, and GROUP BY clauses. Use composite indexes with columns ordered by selectivity (most selective first) matching your query patterns. Use partial indexes for queries on subsets (`WHERE status = 'active'`). Use covering indexes to avoid table lookups for frequently queried column combinations. Monitor slow query logs weekly and run `EXPLAIN ANALYZE` on the top 10 slowest queries. Remove unused indexes as they slow writes.
4. **Implement Multi-Layer Caching** — Design caching in layers: CDN cache (static assets, public API responses), application-level cache (Redis/Memcached for computed results, session data, rate limit counters), and in-process cache (local memory for hot configuration, small lookup tables). Define TTLs based on data freshness requirements. Implement cache invalidation via explicit delete-on-write for mutable data. Use cache-aside pattern (read: check cache, miss -> query DB -> populate cache) as the default.
5. **Choose the Right Pagination Strategy** — Use cursor-based pagination (keyset pagination) for infinite scroll, real-time feeds, and large datasets — it performs consistently regardless of offset depth. Use offset-based pagination only for small datasets or when users need to jump to arbitrary pages. For cursor pagination, encode the last item's sort key as an opaque cursor. Always enforce a maximum page size (100-500 items). Include total count only when explicitly requested, as `COUNT(*)` can be expensive.
6. **Offload Work to Async Processing** — Move non-latency-critical work out of the request path into background queues: email sending, webhook delivery, report generation, image processing, audit logging, analytics events. Use a reliable message queue (Redis Streams, RabbitMQ, SQS, NATS JetStream) with at-least-once delivery, dead-letter queues for failed jobs, and idempotency keys to handle retries safely. Return a 202 Accepted with a status polling URL for long-running operations.
7. **Implement Rate Limiting with Appropriate Algorithms** — Use token bucket or sliding window algorithms for API rate limiting. Apply limits per API key, per user, and per IP at different thresholds. Return `429 Too Many Requests` with `Retry-After` and `X-RateLimit-Remaining` headers. Implement rate limiting at the API gateway level for efficiency. Use Redis for distributed rate limiting across multiple application instances. Set separate limits for different endpoint classes (read vs. write, bulk vs. single).
8. **Configure Load Balancing for Optimal Distribution** — Use least-connections or weighted round-robin algorithms rather than simple round-robin to handle uneven request durations. Enable health checks with appropriate intervals (10-30 seconds) and failure thresholds (2-3 consecutive failures). Configure connection draining for graceful shutdown during deployments. For WebSocket or long-polling, use sticky sessions or connection-aware routing. Monitor per-backend request distribution and latency to detect imbalances.
9. **Enable Response Compression** — Compress API responses with gzip or Brotli for payloads over 1 KB. Brotli achieves 15-25% better compression than gzip at comparable CPU cost. Configure compression at the reverse proxy or CDN level rather than in application code. Set appropriate `Content-Encoding` and `Vary: Accept-Encoding` headers. Skip compression for already-compressed formats (images, videos, pre-compressed archives). For high-throughput APIs, benchmark CPU impact and consider pre-compressed static responses.
10. **Use Database Read Replicas** — Route read-heavy queries (reports, dashboards, search, listing endpoints) to read replicas to offload the primary database. Be aware of replication lag (typically 10-100ms) and route only eventually-consistent reads to replicas. Keep all writes and reads-after-writes on the primary. Implement replica-aware connection management in your ORM or data access layer. Monitor replication lag and fall back to primary when lag exceeds your consistency threshold.
11. **Set Connection Timeouts at Every Layer** — Configure timeouts for every external call: connection timeout (1-5 seconds), read/write timeout (5-30 seconds based on expected operation time), and idle timeout. Set overall request timeout at the API gateway (30-60 seconds). Implement circuit breakers (Hystrix pattern) that open after consecutive failures and periodically probe for recovery. Never use infinite timeouts. Log timeout events with destination, duration, and operation context for debugging.
12. **Profile Before Optimizing** — Use profiling tools to identify actual bottlenecks before optimizing: CPU profilers (perf, flamegraphs, pprof), memory profilers (valgrind, heaptrack), database query analyzers (pg_stat_statements, slow query log), and distributed tracing for cross-service latency. Optimize the top 3 bottlenecks that account for 80% of latency. Never optimize based on intuition alone; profile under production-like load to surface real hotspots.
13. **Optimize Serialization and Deserialization** — For high-throughput APIs, serialization can consume 10-30% of CPU. Use efficient serializers: `serde` in Rust, `orjson` in Python, `simdjson` for parsing. For internal service-to-service communication, consider binary formats (Protocol Buffers, MessagePack, FlatBuffers) which are 2-10x faster and smaller than JSON. Stream large responses instead of buffering the entire payload in memory. Use `Content-Type` negotiation to serve the optimal format per client.
14. **Implement Graceful Degradation Under Load** — Design services to shed load gracefully rather than crash: implement admission control that rejects requests with 503 when queue depth exceeds capacity, prioritize critical endpoints over non-critical ones, serve cached stale data when backends are slow (with `Cache-Control: stale-if-error`), and disable expensive features (full-text search, recommendations) under high load. Load test regularly to find breaking points and tune thresholds.
