# Confidential Computing

When working with confidential computing:

1. Choose the right TEE technology based on your threat model: Intel SGX for application-level enclaves with small TCB, AMD SEV-SNP for full-VM memory encryption with integrity, and ARM TrustZone for embedded/mobile secure-world isolation.
2. Use Azure Confidential VMs (DCasv5/ECasv5 series with AMD SEV-SNP or DCsv3 with Intel SGX) for lift-and-shift scenarios where the entire VM memory is encrypted; no application code changes are needed for VM-level confidential computing.
3. Implement remote attestation to verify that code is running inside a genuine TEE before sending sensitive data; use Microsoft Azure Attestation (MAA) or Intel's DCAP attestation to validate hardware evidence (quote/report) cryptographically.
4. Develop SGX enclaves using the Open Enclave SDK (`oe_create_enclave`, `oe_call_enclave_function`) which provides a portable abstraction across Intel SGX and ARM TrustZone; define ECALL/OCALL interfaces in EDL files to minimize the enclave attack surface.
5. Minimize the trusted computing base by keeping only cryptographic operations and sensitive data processing inside the enclave; move I/O, networking, and non-sensitive logic to the untrusted host application to reduce enclave memory pressure and audit scope.
6. Deploy confidential containers on AKS with `confcom` add-on or on ACI with `--sku Confidential`; use security policies generated by `az confcom` tooling to define allowed container images, environment variables, and mount points enforced by the TEE.
7. Encrypt data before it leaves the TEE using enclave-generated keys, and seal data to the enclave identity using `oe_seal_data` (Open Enclave) or `sgx_seal_data` (Intel SDK) for persistent storage that only the same enclave can unseal.
8. Handle enclave memory constraints (EPC for SGX is typically 64-512 MB) by streaming data through the enclave in chunks rather than loading entire datasets; EPC paging degrades performance significantly, so profile with `sgx_emmt` to monitor usage.
9. Use AMD SEV-SNP's vTPM and VMPL (Virtual Machine Privilege Levels) to establish trust chains in confidential VMs; the firmware-measured boot sequence and vTPM attestation enable Measured Boot verification without modifying the guest OS.
10. Implement secure key release workflows where an HSM or key vault releases encryption keys only after verifying a valid attestation token; Azure Key Vault's Secure Key Release and MAA integration automate this pattern for confidential workloads.
11. Test confidential computing applications using simulation mode (`OE_SIMULATE=1` for Open Enclave, `SGX_MODE=SIM` for Intel SDK) during development; simulation provides functional correctness testing without requiring TEE hardware, but always validate on real hardware before production.
12. Address side-channel risks by using constant-time cryptographic implementations inside enclaves, avoiding secret-dependent memory access patterns and branching; apply compiler hardening flags (`-mindirect-branch=thunk`, LVI mitigations) and stay current with microcode updates.
