# Hard Problem Decomposition and Solving

When tackling complex, ambiguous, or multi-file engineering problems:

1. **Always Decompose Before Coding** — Never write code as the first action on a hard problem. First produce a written decomposition: identify the problem's core invariant, list the major subproblems, identify the dependencies between them, and sequence the subproblems in topological order. The decomposition itself is the primary deliverable of the first step — share it with the user for validation before coding begins.
2. **Surface Assumptions Explicitly** — Before decomposing, enumerate every assumption the solution will rely on. Format each assumption as a falsifiable statement: "Assumes the database schema has not changed since the last migration", "Assumes all callers of `getUserById` pass a non-null ID". Surface assumptions to the user immediately; a wrong assumption invalidates the entire solution. Track which assumptions were confirmed, which were rejected, and which remain unverified.
3. **Start with the Smallest Verifiable Unit** — After decomposing, begin implementation with the smallest unit whose correctness can be independently verified (a single function, a single API endpoint, a single data transformation). Write a test for it before or alongside the implementation. Verify the unit works in isolation before integrating. This hypothesis-testing approach catches fundamental flaws early before they are amplified across multiple files.
4. **Generate Clarifying Questions for Short Specs** — When the problem spec is fewer than 50 characters (e.g., "fix the login bug", "refactor auth"), the spec is too ambiguous to act on. Generate 3–5 targeted clarifying questions covering: the desired behavior, the acceptable solution constraints, the definition of done, and any known gotchas. Prioritize questions whose answers would change the solution approach. Do not guess and proceed — ambiguous specs produce solutions that miss the mark.
5. **Topological Change Ordering** — When a problem requires changes across multiple files, derive the dependency graph of those changes and apply them in topological order (dependencies first, dependents last). This ensures the codebase is always in a compilable/runnable state between commits. Never apply changes in alphabetical file order or arbitrary order — dependency violations will cascade into confusing errors.
6. **Constraint Identification** — Identify hard constraints (must not break the public API, must pass existing tests, must complete in <100ms) and soft constraints (prefer immutability, prefer functional style, prefer minimal diff) before designing the solution. Hard constraints are gates: any solution violating a hard constraint is discarded. Soft constraints are tiebreakers between otherwise equivalent solutions.
7. **Incremental Verification Checkpoints** — Divide the implementation into checkpoints where the full test suite (or a relevant subset) can be run. Run tests at each checkpoint and halt if failures appear before proceeding to the next step. A hard problem solved in one giant commit with a test suite run only at the end is an anti-pattern — failures are hard to isolate. Commit passing checkpoints as intermediate commits to preserve working state.
8. **Post-Solution Review** — After completing the solution, review the original decomposition against what was actually built. Note any subproblems that were harder or easier than expected, any assumptions that turned out to be wrong, and any scope creep that occurred. Write a brief retrospective note (3–5 sentences) alongside the final commit message. This retrospective feeds into future decomposition quality for similar problems.
