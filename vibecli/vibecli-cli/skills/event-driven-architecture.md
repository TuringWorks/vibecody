# Event-Driven Architecture Patterns

When designing event-driven systems with event sourcing, CQRS, and messaging patterns:

1. **Event Sourcing** — Store state changes as an immutable, append-only sequence of domain events rather than overwriting current state. Reconstruct entity state by replaying events from the beginning or from a snapshot. Use event versioning (schema registry) to handle evolution, and compact old events into periodic snapshots to bound replay time.
2. **CQRS Separation** — Split your system into a command side (handles writes, validates business rules, emits events) and a query side (builds read-optimized projections from events). This allows independent scaling, different storage engines per side (e.g., PostgreSQL for writes, Elasticsearch for reads), and specialized query models without compromising write integrity.
3. **Saga Pattern for Distributed Transactions** — Coordinate multi-service transactions using sagas: a sequence of local transactions where each step publishes an event triggering the next. Implement compensating transactions for rollback on failure. Choose choreography (services react to events independently) for simple flows and orchestration (central coordinator) for complex multi-step workflows.
4. **Transactional Outbox** — Avoid dual-write problems by writing domain events to an outbox table within the same database transaction as the business state change. A separate process (poller or CDC connector) reads the outbox and publishes events to the message broker. This guarantees at-least-once delivery without distributed transactions.
5. **Change Data Capture with Debezium** — Deploy Debezium connectors to stream database changes (INSERT, UPDATE, DELETE) as events to Kafka. Use log-based CDC to capture changes without modifying application code, configure snapshot mode for initial data load, and set up schema history topics to track table schema evolution over time.
6. **Event Schema Evolution** — Define event schemas in a registry (Confluent Schema Registry, AWS Glue) using Avro, Protobuf, or JSON Schema. Enforce backward/forward compatibility rules: add optional fields (backward compatible), never remove required fields, never change field types. Version event types and support multiple versions in consumers during migration.
7. **Dead-Letter Handling** — Route events that fail processing after retries to a dead-letter queue/topic. Include the original event, error details, retry count, and timestamp. Build tooling to inspect, replay, or discard dead-lettered events. Set up alerts on DLQ depth and review failures daily to identify systematic issues versus transient errors.
8. **Exactly-Once Processing** — Achieve effective exactly-once semantics through idempotent consumers: track processed event IDs in the consumer's database and skip duplicates. Use Kafka's transactional producer + consumer offsets for end-to-end exactly-once within Kafka. Design event handlers to be naturally idempotent where possible (set operations, upserts).
9. **Event Storming for Domain Discovery** — Run collaborative Event Storming workshops to discover domain events, commands, aggregates, and bounded contexts before writing code. Use orange stickies for domain events, blue for commands, yellow for aggregates, and pink for hotspots. The resulting event flow becomes your system architecture blueprint.
10. **Choreography vs Orchestration** — Use choreography (services publish and subscribe to events independently) when services are loosely coupled and flows are simple. Use orchestration (a central coordinator manages the workflow) when you need visibility into multi-step processes, complex branching logic, or timeout handling. Many systems benefit from a hybrid approach.
11. **Event-Driven Projections** — Build specialized read models by subscribing to event streams and maintaining denormalized views. Run multiple projections from the same event stream for different use cases (dashboard aggregations, search indexes, notification triggers). Make projections rebuildable: delete and replay from the event store when the schema changes.
12. **Backpressure and Flow Control** — Implement backpressure to prevent fast producers from overwhelming slow consumers. Use bounded queues with reject/drop policies, consumer group lag monitoring (Kafka consumer lag), rate limiting at the producer, and auto-scaling consumers based on queue depth. Alert when consumer lag exceeds acceptable thresholds and investigate root causes.
