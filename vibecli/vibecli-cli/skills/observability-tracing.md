# Observability: Distributed Tracing

When implementing distributed tracing across services:

1. **Use OpenTelemetry as Your Instrumentation Standard** — Adopt the OpenTelemetry SDK as your single instrumentation layer regardless of backend (Jaeger, Zipkin, Tempo, Datadog, Honeycomb). OTel provides vendor-neutral APIs, auto-instrumentation for common libraries, and exporters for every major backend. Pin to a stable SDK version across all services and update them together to avoid protocol mismatches.
2. **Create Meaningful Spans** — Create a span for every significant unit of work: inbound HTTP/gRPC requests, outbound service calls, database queries, cache operations, message queue publish/consume, and CPU-intensive computations. Name spans as `<verb> <resource>` (e.g., `GET /users`, `query users_table`, `publish order.created`). Avoid creating spans for trivial operations (in-memory lookups, simple conditionals) as they add noise and overhead.
3. **Propagate W3C Trace Context Headers** — Use the W3C `traceparent` and `tracestate` headers for cross-service context propagation. Configure your HTTP clients, message queue producers, and gRPC interceptors to inject these headers automatically. On the receiving side, extract the context before creating new spans so child spans are correctly linked to the parent trace. Test propagation across all service boundaries, including async message queues where headers travel in message metadata.
4. **Implement Head-Based Sampling for Cost Control** — At high traffic volumes, tracing 100% of requests is prohibitively expensive. Implement head-based sampling at the entry point: sample 1-10% of normal traffic, 100% of error responses, and 100% of requests exceeding your SLO latency threshold. Use a consistent sampling decision based on the trace ID so all spans in a sampled trace are either fully collected or fully dropped.
5. **Add Tail-Based Sampling for High-Value Traces** — Deploy a tail-sampling processor (OpenTelemetry Collector with `tail_sampling` processor) that buffers complete traces and makes sampling decisions after seeing all spans. Keep 100% of traces with errors, high latency (>p99), or specific attributes (e.g., premium tier customers). This captures the interesting traces that head-based sampling would miss, at the cost of requiring a stateful collector with sufficient memory.
6. **Correlate Traces with Logs and Metrics** — Inject `trace_id` and `span_id` into every log line emitted within an active span context. In your logging library, configure automatic extraction from the current span context. In Grafana, configure Loki-to-Tempo and Tempo-to-Loki data source correlations. This lets you jump from a slow trace to the exact log lines and from an error log to the full distributed trace.
7. **Use Baggage for Cross-Cutting Context** — Use OpenTelemetry baggage to propagate business context (tenant ID, feature flag values, deployment version, user tier) across service boundaries without changing API contracts. Baggage travels in HTTP headers alongside trace context. Extract baggage values into span attributes at each service for filtering. Be mindful of header size limits; keep baggage under 8 KB total.
8. **Leverage Auto-Instrumentation Libraries** — Before writing manual instrumentation, enable auto-instrumentation for your framework and libraries: `opentelemetry-instrumentation-*` packages for Python, `@opentelemetry/instrumentation-*` for Node.js, Java agent for JVM services, `tracing` crate with `opentelemetry` layer for Rust. Auto-instrumentation covers HTTP servers/clients, database drivers, and messaging libraries with zero code changes.
9. **Add Semantic Attributes to Spans** — Follow OpenTelemetry semantic conventions for span attributes: `http.method`, `http.status_code`, `http.url`, `db.system`, `db.statement`, `messaging.system`, `messaging.destination`. Add custom attributes for business context: `order.id`, `customer.tier`, `feature.flag`. These attributes power trace filtering, grouping, and analysis in your backend. Never add high-cardinality unbounded values as attributes.
10. **Design for Trace Visualization** — Structure your spans to tell a readable story in the trace waterfall view. Use span events (annotations) to mark significant milestones within a span (e.g., "cache miss", "retry attempt 2"). Use span links to connect causally related but not parent-child traces (e.g., a batch job triggered by a cron linking to the original API request that queued the work). Set span status to ERROR with a description when operations fail.
11. **Manage Performance Overhead** — Tracing adds 1-5% overhead per instrumented call. Measure the actual impact in your environment with benchmarks. Reduce overhead by: using async batch exporters (not synchronous), limiting span attribute count to 32-64, capping span event count, using OTLP/gRPC export (more efficient than HTTP/JSON), and configuring appropriate batch sizes and export intervals (5-30 seconds).
12. **Propagate Context Across Async Boundaries** — In event-driven architectures, trace context must survive message queues, pub/sub systems, and scheduled jobs. Serialize trace context into message headers (Kafka, RabbitMQ, SQS) or metadata fields. On the consumer side, create a new span linked to the producer span. For cron jobs or scheduled tasks, create a new root trace but link it to the triggering context if available.
13. **Set Up Trace-Based Testing in CI** — In integration tests, capture traces emitted during test execution using an in-memory exporter. Assert on trace structure: verify expected spans exist, parent-child relationships are correct, required attributes are present, and no error statuses appear on success paths. This catches instrumentation regressions and ensures propagation works before deployment.
14. **Monitor Your Tracing Pipeline Health** — Track the health of the tracing pipeline itself: spans generated per second, spans dropped (exporter queue full, network errors), collector CPU/memory usage, and backend ingestion lag. Alert when the drop rate exceeds 5% or collector memory exceeds 80%. A broken tracing pipeline gives you false confidence that everything is fine when you are actually flying blind.
