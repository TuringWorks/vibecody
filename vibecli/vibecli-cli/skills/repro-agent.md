# Reproducible and Deterministic Agent Sessions

When building agents that must be reproducible, debuggable, or validated in CI:

1. **Always Snapshot Before Long Sessions** — Before starting any agent session expected to run longer than 5 minutes or involving more than 10 tool calls, capture a hermetic session snapshot. The snapshot must be completed and persisted before the first tool call is made. This ensures that if the session produces unexpected output, it can be replayed from the exact starting state without requiring manual reconstruction of the environment.
2. **Mark Non-Deterministic Tools Explicitly** — Tag each tool in the agent's tool registry as deterministic or non-deterministic. Non-deterministic tools: current datetime, random number generation, external API calls with varying responses, file system reads of frequently-changing files, network requests. In a reproducible session, non-deterministic tools are intercepted and their responses are recorded on first run and replayed verbatim on subsequent runs. Never replay a non-deterministic tool's live output — always use the recorded response.
3. **Session Diff for Debugging Behavior Changes** — When an agent session produces a different output than a previous run of the same task, generate a structured diff of: the tool call sequence (added/removed/reordered calls), tool inputs (parameter changes), tool outputs (response changes), and the final answer. Display the diff in a human-readable format highlighting the first point of divergence. This pinpoints whether the behavior change was caused by a model change, a tool response change, or an environment change.
4. **CI Gate Threshold: 100% Match** — In CI pipelines validating agent reproducibility, require 100% match between the recorded session and the replay session. Partial matches are not acceptable for reproducibility CI gates — a 99% match means the agent is non-deterministic. If 100% match cannot be achieved due to inherently non-deterministic operations (e.g., timestamps in output), those fields must be explicitly excluded from comparison via a determinism exclusion list maintained in the test configuration.
5. **Hermetic Snapshot Components** — A hermetic snapshot must include: (1) lock files for all package managers (Cargo.lock, package-lock.json, requirements.txt with pinned hashes), (2) all environment variables relevant to the agent (captured at snapshot time, with secrets replaced by reference keys), (3) the random seed used for any stochastic operations, (4) the model version and temperature settings, (5) the tool registry state (which tools are available and at what versions), and (6) a content-addressed snapshot of all input files the agent will read. Snapshots missing any of these components cannot guarantee reproducibility.
6. **Replay Fidelity Modes** — Support two replay fidelity modes: `strict` (all tool responses replayed verbatim, model calls use recorded outputs) and `partial` (tool responses replayed, model re-executes with the same inputs). Strict mode produces identical output for debugging; partial mode validates that the model reaches the same conclusions given the same tool responses, testing model stability independently of environment changes. Always use strict mode for CI gates; partial mode is for model upgrade validation.
7. **Snapshot Storage and Retention** — Store session snapshots in a content-addressed store (hash of the hermetic snapshot components). Retain snapshots for: 30 days for normal sessions, 90 days for sessions that produced bugs or regressions, and indefinitely for sessions used as CI baseline references. Snapshots referenced by CI baselines must be explicitly protected from expiry. Include a `snapshot_metadata.json` alongside each snapshot: task description, agent version, model, creation timestamp, and replay count.
8. **Flakiness Attribution** — When a session fails the reproducibility CI gate, attribute the source of flakiness automatically: compare the first divergence point in the session diff against the tool's determinism tag. If a deterministic tool produced different output, investigate the tool for environment-dependent behavior (hidden file reads, global state, OS-level randomness). If a non-deterministic tool was not intercepted, fix the interception layer. Track flakiness rate per tool over time; tools with >1% flakiness rate must be reviewed and either fixed or downgraded to non-deterministic tagging.
