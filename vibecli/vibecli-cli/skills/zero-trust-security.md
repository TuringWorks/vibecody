# Zero Trust Architecture

When implementing zero trust security principles:

1. **Never Trust, Always Verify** — Authenticate and authorize every request regardless of network location. Internal services are not inherently trusted; a request from inside the VPC receives the same scrutiny as one from the public internet. Validate identity tokens on every hop, not just at the edge. Assume the network is always hostile and compromised.
2. **Microsegmentation** — Replace flat network trust zones with fine-grained segments that restrict lateral movement. Use Kubernetes NetworkPolicies, cloud security groups, or software-defined networking to enforce per-service communication rules. Default to deny-all and explicitly allow only required service-to-service paths. Regularly audit and prune stale rules.
3. **Identity-Aware Proxy** — Deploy an identity-aware proxy (Google IAP, Pomerium, Teleport) in front of internal applications. Users authenticate via SSO/MFA before reaching any application, eliminating the need for VPN access. The proxy enforces per-application access policies based on user identity, device posture, and context (time, location, risk score).
4. **mTLS Everywhere** — Encrypt and authenticate all service-to-service communication using mutual TLS. Use short-lived certificates (24 hours or less) rotated automatically. Deploy a service mesh (Istio, Linkerd) or certificate manager (cert-manager, Vault PKI) to handle certificate issuance and rotation without application code changes.
5. **SPIFFE/SPIRE Identity** — Adopt the SPIFFE standard for workload identity, assigning each service a SPIFFE ID (spiffe://trust-domain/path) with X.509 or JWT SVIDs. Deploy SPIRE as the identity provider, configure attestors for your platform (Kubernetes, AWS, bare metal), and use SPIFFE IDs as the basis for authorization decisions across service boundaries.
6. **Policy Engines: OPA and Cedar** — Externalize authorization logic into a dedicated policy engine. Use Open Policy Agent (OPA) with Rego policies for Kubernetes admission control, API authorization, and data filtering. Evaluate Cedar for strongly-typed, analyzable policies in application-level authorization. Store policies in Git, test with unit test frameworks, and deploy via CI/CD.
7. **Continuous Verification** — Verify identity and authorization not just at connection time but continuously throughout the session. Re-evaluate access policies periodically (every few minutes), check for revoked credentials, monitor for anomalous behavior (unusual API patterns, geographic impossibility), and terminate sessions that no longer meet policy requirements.
8. **Least Privilege Access** — Grant the minimum permissions required for each identity, scoped by time, resource, and action. Use just-in-time (JIT) access for elevated privileges with automatic expiration. Implement service accounts with narrowly scoped IAM roles (one role per service, not shared), and review access grants quarterly to remove accumulated permissions.
9. **BeyondCorp Model** — Implement Google's BeyondCorp approach: move access controls from the network perimeter to individual users and devices. Assess device trust (OS patched, disk encrypted, endpoint agent running) as part of every access decision. Combine user identity, device posture, and request context into a trust score that determines access level.
10. **Device Trust and Posture** — Evaluate device security posture before granting access: verify OS version, encryption status, endpoint detection and response (EDR) agent health, and certificate validity. Use MDM signals or device attestation services. Downgrade access level (e.g., read-only instead of read-write) for devices that fail posture checks rather than blocking entirely.
11. **Audit Logging and Forensics** — Log all authentication attempts, authorization decisions, and access events to a tamper-evident audit log. Include timestamp, identity, action, resource, decision (allow/deny), and policy version. Stream logs to a SIEM (Splunk, Elastic Security), set up correlation rules for anomaly detection, and retain logs for compliance-mandated periods.
12. **Encrypted Data at Rest and in Transit** — Encrypt all data in transit (TLS 1.3, mTLS) and at rest (AES-256, envelope encryption with KMS). Use customer-managed encryption keys where compliance requires it, implement key rotation schedules, and ensure encryption extends to backups, logs, and temporary storage. Audit encryption coverage regularly to find gaps.
