# Frontend Performance Optimization

When optimizing frontend performance and Core Web Vitals:

1. **Optimize Largest Contentful Paint (LCP)** — Target LCP under 2.5 seconds. Identify your LCP element (usually a hero image, heading, or video poster) and optimize its loading path: preload the resource with `<link rel="preload">`, inline critical CSS to avoid render-blocking stylesheets, use `fetchpriority="high"` on the LCP image, serve images from a CDN geographically close to users, and eliminate any redirect chains on the document URL. Server response time (TTFB) directly impacts LCP, so optimize backend response time to under 200ms.
2. **Minimize Cumulative Layout Shift (CLS)** — Target CLS under 0.1. Always set explicit `width` and `height` attributes (or CSS `aspect-ratio`) on images and videos so the browser reserves space before loading. Use `min-height` on dynamic content containers (ads, embeds, lazy-loaded sections). Avoid inserting content above existing visible content. Use CSS `contain: layout` on components that will update. Load web fonts with `font-display: swap` combined with `size-adjust` and `ascent-override` to match fallback font metrics.
3. **Implement Code Splitting Strategically** — Split your JavaScript bundle by route (each page loads only its code), by component (heavy components like editors, charts, maps load on demand), and by vendor (third-party libraries in a separate chunk for better caching). Use dynamic `import()` for route-based splitting with React.lazy/Suspense, Vue async components, or framework-specific APIs. Aim for an initial JavaScript payload under 200 KB compressed.
4. **Lazy Load Below-the-Fold Content** — Use `loading="lazy"` on images and iframes below the fold. Never lazy-load the LCP image. For JavaScript-heavy components (maps, charts, comment sections), load them when they enter the viewport using `IntersectionObserver`. Implement a loading skeleton or placeholder to prevent layout shift. Prefetch likely-needed resources on hover or when the user scrolls near them.
5. **Optimize Images Aggressively** — Serve images in modern formats: WebP (30% smaller than JPEG) or AVIF (50% smaller) with JPEG/PNG fallbacks via `<picture>` and `<source>`. Use responsive images with `srcset` and `sizes` to serve appropriately sized images per viewport. Compress images to 80% quality (usually imperceptible). Use a CDN with on-the-fly image transformation (Cloudflare Images, Imgix, Cloudinary) rather than pre-generating every variant. Limit image dimensions to 2x the display size for retina support.
6. **Analyze and Reduce Bundle Size** — Run bundle analysis regularly (`webpack-bundle-analyzer`, `source-map-explorer`, or Vite's `rollup-plugin-visualizer`). Identify and eliminate: duplicate dependencies (multiple versions of the same library), unused exports (ensure tree shaking works by using ES modules), heavy libraries with lighter alternatives (replace `moment.js` with `date-fns` or `dayjs`, `lodash` with individual imports), and polyfills for APIs your target browsers already support.
7. **Implement Effective Tree Shaking** — Ensure your bundler can eliminate dead code: use ES module `import`/`export` (not CommonJS `require`), mark packages as side-effect-free in `package.json` (`"sideEffects": false`), avoid barrel files that re-export entire directories, and import specific functions (`import { debounce } from 'lodash-es'` not `import _ from 'lodash'`). Verify tree shaking works by checking the bundle output for removed exports.
8. **Configure CDN Caching and Cache Busting** — Serve static assets with content-hashed filenames (`app.a1b2c3.js`) and long Cache-Control headers (`max-age=31536000, immutable`). Serve the HTML document with `no-cache` or short `max-age` so updates propagate immediately. Use a CDN with edge caching and configure appropriate `Vary` headers. Implement `stale-while-revalidate` for API responses that can tolerate brief staleness.
9. **Extract and Inline Critical CSS** — Identify the CSS required to render above-the-fold content and inline it in the `<head>`. Load remaining CSS asynchronously with `<link rel="preload" as="style" onload="this.rel='stylesheet'">`. Tools like `critical` or `critters` (Webpack plugin) automate this extraction. Keep inlined CSS under 14 KB (fits in the first TCP round trip). Remove unused CSS with PurgeCSS or the framework equivalent.
10. **Optimize Font Loading** — Use `font-display: swap` to show text immediately with a fallback font, then swap when the custom font loads. Subset fonts to include only the characters you use (latin, latin-ext). Prefer `woff2` format (30% smaller than woff). Self-host fonts instead of loading from Google Fonts to eliminate the extra DNS lookup and connection. Preload the primary font with `<link rel="preload" href="font.woff2" as="font" type="font/woff2" crossorigin>`.
11. **Implement Service Worker Caching** — Use a service worker with Workbox for intelligent caching strategies: Cache-First for static assets (JS, CSS, images), Network-First for API calls, Stale-While-Revalidate for semi-dynamic content. Precache the app shell for instant repeat visits. Implement offline fallback pages. Keep the service worker update flow smooth with `skipWaiting()` and `clients.claim()` to avoid stale content.
12. **Use Preload, Prefetch, and Preconnect** — Use `<link rel="preconnect">` for critical third-party origins (API servers, CDNs, font servers) to establish connections early. Use `<link rel="preload">` for resources needed in the current navigation (hero image, critical font, above-the-fold data API). Use `<link rel="prefetch">` for resources likely needed on the next navigation (next page's JS bundle). Avoid over-prefetching on low-bandwidth connections; check `navigator.connection.effectiveType`.
13. **Set and Enforce Performance Budgets** — Define budgets for key metrics: total JavaScript under 300 KB compressed, total CSS under 50 KB, LCP under 2.5s, CLS under 0.1, Time to Interactive under 3.5s. Enforce budgets in CI with Lighthouse CI, `bundlesize`, or `size-limit`. Fail the build when budgets are exceeded. Review and adjust budgets quarterly based on actual user data from the Chrome User Experience Report (CrUX) or Real User Monitoring (RUM).
14. **Monitor Real User Performance** — Collect Core Web Vitals from real users using the `web-vitals` library or a RUM provider (Datadog RUM, SpeedCurve, Sentry Performance). Segment by device type, connection speed, geography, and page. Synthetic tests (Lighthouse, WebPageTest) catch regressions in CI but do not represent real-world variability. Set up alerts when p75 LCP, CLS, or INP exceed thresholds and investigate regressions within one sprint.
