# Web3 DeFi Development

When building decentralized finance protocols and applications:

1. **Design AMM Pools with Correct Invariant Math** — Implement constant product (`x * y = k`) for Uniswap V2-style pools, concentrated liquidity with tick-based ranges for V3-style efficiency, or stable swap curves (StableSwap invariant) for pegged asset pairs. Use fixed-point arithmetic libraries (PRBMath, ABDKMath64x64) to avoid precision loss — never use floating-point in smart contracts. Implement proper rounding (round against the trader, in favor of the pool) to prevent dust-level exploits. Test invariant preservation across thousands of random swap sequences with fuzzing tools like Echidna or Foundry's fuzz testing.
2. **Implement Liquidity Pool Mechanics Safely** — Track LP token supply and individual positions accurately using shares-based accounting (mint LP tokens proportional to contributed liquidity relative to total pool value). Handle the first deposit edge case by minting to a dead address to prevent share price manipulation. Implement minimum liquidity locks to prevent total withdrawal. For concentrated liquidity, store position ranges and track fees accrued per tick. Emit events for all deposits, withdrawals, and fee claims to enable off-chain indexing with The Graph or custom indexers.
3. **Build Yield Farming with Sustainable Tokenomics** — Design reward distribution using proven models: MasterChef-style (rewards per block distributed proportionally to staked amounts), gauge voting (veToken model for directing emissions), or real yield (protocol revenue sharing instead of inflationary tokens). Implement reward calculation with accumulated reward per share pattern to avoid gas-intensive loops. Include vesting schedules and lock-up periods to reduce sell pressure. Model token emission schedules for 3-5 year sustainability. Use time-weighted average balances to prevent just-in-time staking for rewards.
4. **Secure Flash Loan Implementations** — For flash loan providers, enforce the invariant that borrowed amount plus fee must be returned within the same transaction — check the balance at the end, not the beginning. Implement reentrancy guards on all state-changing functions. For protocols vulnerable to flash loan attacks, use time-weighted average prices (TWAPs) from Uniswap V3 instead of spot prices for oracle data. Add minimum hold period requirements for governance voting power. Test all protocol functions with flash loan wrappers to identify manipulation vectors.
5. **Design Lending and Borrowing Protocols Carefully** — Implement interest rate models (utilization-based curves like Compound's jump rate model) that incentivize optimal pool utilization. Use exchange rates or share-based accounting for supply/borrow positions. Implement liquidation mechanisms with liquidation bonuses (5-10%) to incentivize liquidators. Set conservative collateral factors (60-80%) based on asset volatility and liquidity. Use Chainlink or Pyth oracles for price feeds with staleness checks and circuit breakers. Implement reserve factors (10-20%) to build a protocol safety fund.
6. **Build Governance Systems That Resist Attacks** — Implement time-locked governance with proposal lifecycle: submission (with minimum token threshold), voting period (3-7 days), timelock delay (24-48 hours), and execution. Use vote-escrowed tokens (veTokens) to align long-term holder interests — longer lock periods grant more voting power. Implement quorum requirements (4-10% of total supply) and supermajority thresholds for critical changes. Add a guardian role that can cancel malicious proposals during the timelock period. Snapshot voting off-chain with on-chain execution reduces gas costs while maintaining security.
7. **Design Token Economics with Long-Term Viability** — Model token supply dynamics: fixed supply with deflationary mechanisms (burns on fees), controlled inflation with emission schedules, or elastic supply (rebase tokens). Allocate tokens across stakeholders with vesting: team (15-20%, 4-year vest with 1-year cliff), investors (15-20%, 2-year vest), community/ecosystem (40-50%, distributed via farming and grants), treasury (10-15%). Simulate scenarios: what happens at 10x and 0.1x token price? Do incentives still align? Avoid designs where the protocol only works when the token price goes up.
8. **Integrate Oracles with Defense in Depth** — Never rely on a single oracle source. Use Chainlink as the primary oracle with Uniswap V3 TWAP as a fallback. Implement price deviation checks: if the oracle price deviates more than 5-10% from the TWAP, pause the protocol or use the more conservative price. Add staleness checks (reject prices older than the heartbeat interval). Implement circuit breakers that pause liquidations during extreme price movements. For exotic assets without Chainlink feeds, use multiple DEX TWAPs with longer observation windows (30 minutes to 1 hour).
9. **Protect Against MEV Exploitation** — Implement MEV protection strategies: use commit-reveal schemes for sensitive operations (limit orders, auction bids), integrate with Flashbots Protect or MEV blocker RPC for transaction privacy, design with MEV-resistant ordering (batch auctions like CowSwap instead of continuous trading), and implement slippage protection with user-defined maximum slippage (default 0.5-1%). For DEX aggregators, use private transaction pools and implement deadline parameters to prevent transaction withholding. Educate users about front-running risks in your documentation.
10. **Follow DeFi Security Audit Patterns** — Get at least two independent audits before mainnet launch from reputable firms. Run a bug bounty program (Immunefi) with bounties proportional to TVL. Use formal verification for critical invariants (Certora, Halmos). Implement the checks-effects-interactions pattern in every external call. Use OpenZeppelin's ReentrancyGuard on all state-modifying external functions. Add pausability for emergency situations with multi-sig control (3-of-5 minimum). Implement timelocks on all parameter changes. Never use `tx.origin` for authorization. Rate limit sensitive operations.
11. **Implement Upgradeable Contracts Carefully** — Use the transparent proxy pattern (OpenZeppelin) or UUPS proxy for upgradeable contracts. Store all state in the proxy contract to preserve data across upgrades. Never change the storage layout order of existing variables — only append new ones. Use storage gaps (`uint256[50] private __gap`) in base contracts to reserve space for future variables. Test upgrades in a fork environment before mainnet deployment. Implement upgrade timelocks (48-72 hours) so users can exit if they disagree with changes. Consider whether upgradeability is even necessary — immutable contracts inspire more trust.
12. **Optimize Gas Costs for DeFi Operations** — Pack storage variables into single 256-bit slots (uint128 + uint128 in one slot). Use `unchecked` blocks for arithmetic where overflow is impossible (loop counters, checked-before math). Cache storage reads in memory variables when accessed multiple times. Use custom errors instead of require strings (saves ~50 gas per revert). Batch operations where possible (multi-swap, multi-claim). Use Solidity 0.8.x with the optimizer at 200-1000 runs. Profile gas with `forge test --gas-report` and target critical paths (swaps, deposits) for sub-100k gas.
13. **Handle Multi-Chain DeFi Deployment** — Deploy to multiple chains with chain-specific considerations: bridge integrations (LayerZero, Axelar, Wormhole) for cross-chain messaging, chain-specific oracle availability, different gas pricing models (EIP-1559 vs. legacy), and varying block times affecting TWAP calculations. Use deterministic deployment (CREATE2) for consistent addresses across chains. Implement chain-specific configuration contracts rather than hardcoding parameters. Test on each chain's testnet independently — EVM compatibility is not identical behavior.
14. **Build Comprehensive DeFi Testing Suites** — Test at four levels: unit tests for individual functions (Foundry or Hardhat), integration tests for multi-contract interactions (fork tests against mainnet state), invariant tests that assert protocol properties hold across random action sequences (Echidna, Foundry invariant testing), and economic simulation tests (agent-based models in Python simulating market conditions). Test edge cases explicitly: zero liquidity, maximum values, rounding boundaries, oracle failure, and reentrancy attempts. Achieve 100% line coverage for core contracts and 100% branch coverage for security-critical functions.
