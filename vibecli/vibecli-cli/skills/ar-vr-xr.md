# AR/VR/XR Development

When building augmented reality, virtual reality, and spatial computing applications:

1. **Use the WebXR Device API for Browser-Based Experiences** — Initialize WebXR sessions with `navigator.xr.requestSession()` specifying the mode: `immersive-vr` for full VR, `immersive-ar` for AR, or `inline` for non-immersive 3D on regular screens. Check feature support with `navigator.xr.isSessionSupported()` before requesting sessions. Use `XRReferenceSpace` types appropriately: `local` for seated experiences, `local-floor` for standing, `bounded-floor` for room-scale, and `unbounded` for world-scale AR. Request only the features you need (`hit-test`, `anchors`, `hand-tracking`, `depth-sensing`) to maximize device compatibility.
2. **Build Spatial Computing Interaction Patterns** — Design interactions that feel natural in 3D space: ray-casting from controllers or gaze direction for pointing, grab-and-release with haptic feedback for manipulation, proximity-based triggers for contextual menus, and spatial gestures (pinch, swipe, palm-up menu) for hand-tracked interfaces. Implement a consistent interaction hierarchy: gaze < controller ray < hand tracking in precision. Provide visual feedback for all interactions (hover highlights, grab indicators, placement previews). Always include fallback input methods — not all headsets support hand tracking.
3. **Optimize the 3D Rendering Pipeline for XR Frame Budgets** — VR demands 90fps (11.1ms per frame) on Quest, 72-120fps depending on device; AR requires similar for smooth overlay rendering. Budget: 4ms for game logic, 3ms for draw calls, 2ms for GPU rendering, 2ms for compositor. Reduce draw calls by batching meshes with shared materials (instanced rendering), using texture atlases, and implementing occlusion culling. Use level-of-detail (LOD) meshes that simplify geometry at distance. Profile with platform tools (Meta Quest Developer Hub, RenderDoc, chrome://gpu) and eliminate any frame that exceeds the budget.
4. **Implement Hand Tracking with Graceful Degradation** — Use the WebXR Hand Input API (`XRHand` with 25 joint poses per hand) or platform SDKs (Meta Hand Tracking SDK, Apple Vision Pro hand tracking). Implement gesture recognition for common actions: pinch (index-thumb distance threshold), point (index extended, others curled), open palm, and fist. Smooth joint positions over 2-3 frames to reduce jitter. Always support controller fallback for devices without hand tracking. Provide visual hand models that match the tracked skeleton. Account for occlusion — hands may be partially visible or lost when outside the tracking volume.
5. **Use Spatial Anchors for Persistent AR Content** — Create anchors at real-world positions using the WebXR Anchors API or platform-specific SDKs (ARCore Cloud Anchors, ARKit World Map, Meta Spatial Anchors). Persist anchor data across sessions for returning users to find content where they left it. Resolve anchors relative to detected planes and surfaces rather than arbitrary coordinates. Handle anchor tracking loss gracefully — show a relocalizing indicator and snap content back when tracking recovers. For shared experiences, use cloud anchors to synchronize spatial positions across multiple users and devices.
6. **Implement Room-Scale Tracking and Guardian Systems** — For room-scale VR, request `bounded-floor` reference space and use the boundary polygon from `XRBoundedReferenceSpace.boundsGeometry` to keep users within their play area. Implement visual warnings (grid fade-in, color shift) when users approach boundaries — do not rely solely on the platform guardian. Design experiences that naturally encourage movement within the tracked volume. For teleportation locomotion, validate destination points against the boundary. Always provide a recenter/reset function for tracking drift.
7. **Meet Performance Targets for Comfortable XR** — Dropped frames cause discomfort and nausea in VR. Target 90fps minimum (72fps absolute minimum on Quest). Minimize motion-to-photon latency below 20ms. Avoid camera-driven locomotion (smooth movement causes motion sickness in 40%+ of users) — prefer teleportation, snap-turn (30-45 degree increments), or physically walking. Reduce vection (perceived self-motion) with stable reference frames, vignetting during movement, and a visible cockpit or nose reference. Test with motion-sickness-prone users during development. Always allow users to adjust comfort settings.
8. **Optimize 3D Assets for XR Platforms** — Target polygon budgets based on platform: mobile VR (Quest): 100k-300k total scene triangles, PC VR: 1M-2M, AR overlays: 50k-100k per object. Use glTF 2.0 as the standard interchange format with Draco compression for meshes and KTX2/Basis Universal for textures. Compress textures to GPU-native formats (ASTC for mobile, BC7 for desktop). Limit texture resolution to 2048x2048 maximum for mobile, use mipmaps always. Bake lighting into lightmaps for static scenes. Use skinned meshes with under 75 bones for animated characters on mobile.
9. **Build Cross-Platform XR with OpenXR** — Use OpenXR as the abstraction layer for multi-platform XR development. OpenXR provides a unified API across Meta Quest, SteamVR, Windows Mixed Reality, and other runtimes. Define interaction profiles for each controller type (Oculus Touch, Valve Index, HP Reverb). Use OpenXR action system rather than raw input — define semantic actions (grab, teleport, menu) and bind them to device-specific inputs. Test on at least two different runtimes to catch platform-specific issues. Use OpenXR extensions selectively (`XR_FB_hand_tracking`, `XR_EXT_eye_gaze_interaction`) knowing they reduce portability.
10. **Implement AR Marker Detection and Image Tracking** — Use WebXR Image Tracking API or platform SDKs (ARCore Augmented Images, ARKit Image Detection) to detect and track printed markers or natural images. Provide high-contrast, feature-rich reference images (minimum 300x300 pixels, avoid repetitive patterns). Pre-register reference images with known physical dimensions for accurate scale. Track detection quality and show confidence indicators to users. Handle multiple simultaneous markers with priority ordering. For marker-less AR, use plane detection and surface estimation to place content on real-world surfaces.
11. **Integrate Spatial Audio for Immersion** — Use the Web Audio API with `PannerNode` (HRTF mode) or platform spatial audio SDKs (Meta Spatial Audio, Steam Audio, Apple Spatial Audio) for 3D sound positioning. Place audio sources at world-space positions that match visual objects. Implement distance attenuation (inverse-square falloff), occlusion (muffling through walls), and room effects (reverb matching the virtual environment). Use ambisonics for environmental audio that responds to head rotation. Keep audio processing under 2ms per frame. Spatial audio dramatically increases presence — prioritize it over visual fidelity when frame budget is tight.
12. **Design Accessible XR Experiences** — XR presents unique accessibility challenges: provide seated mode for users with mobility limitations, offer one-handed interaction alternatives, implement text-to-speech for UI elements, use high-contrast modes and adjustable text sizes, support colorblind-safe palettes with pattern differentiation, allow controller remapping, include subtitles with spatial positioning for audio content, and provide motion comfort options (field-of-view reduction, snap turning, teleport-only movement). Test with assistive technology users. Follow XR Access Initiative guidelines and W3C XR Accessibility User Requirements.
13. **Manage XR Session Lifecycle and Error States** — Handle session states cleanly: request features before session start, listen for `end` event to clean up WebGL resources, handle `visibilitychange` for when the user removes the headset (pause simulation, maintain state), and gracefully recover from tracking loss (`XRFrame.getViewerPose()` returning null). Implement a loading state that works in 3D space. Save user progress automatically — XR sessions end abruptly when batteries die or users remove headsets. Pre-validate device capabilities and show clear messages when required features (hand tracking, plane detection) are unavailable.
14. **Test XR Applications Across Devices and Conditions** — Test on actual hardware, not just emulators — XR performance and interaction feel cannot be evaluated on a flat screen. Test in varied lighting conditions (AR tracking degrades in low light and direct sunlight). Test with different room sizes and tracking space configurations. Verify performance with representative content density, not empty scenes. Use platform-specific profiling tools (OVR Metrics Tool for Quest, SteamVR Frame Timing) to identify CPU/GPU bottlenecks. Conduct user testing with XR novices to catch usability issues that experienced developers overlook.
