# Developer Experience (DX) Best Practices

When designing developer tools, APIs, and platforms:

1. **Write API Documentation with Executable Examples** — Publish OpenAPI/Swagger specifications for every REST API and generate interactive documentation (Swagger UI, Redoc, Stoplight). Include runnable examples for every endpoint with realistic request/response payloads, not lorem ipsum. Show error responses alongside success responses. Provide copy-pasteable `curl` commands and code snippets in the top 3-5 languages your users write. Validate that all examples work in CI by executing them against a test environment.
2. **Design SDKs That Follow Language Idioms** — Each SDK should feel native to its language: use async/await in JavaScript, Result types in Rust, context.Context in Go, type hints in Python. Never generate SDKs solely from OpenAPI specs without human review — generated code often has poor ergonomics. Provide sensible defaults (timeouts, retries, base URL), automatic authentication handling, and typed request/response objects. Include inline documentation with usage examples on every public method.
3. **Build a Self-Service Developer Portal** — Create a single entry point where developers find: getting started guides (5-minute quickstart), API reference, SDK downloads, authentication setup, environment provisioning, status page, and support channels. Use docs-as-code (Markdown in a git repo, built with Docusaurus, MkDocs, or Nextra) so developers can submit corrections via pull request. Track documentation coverage and page engagement to identify gaps.
4. **Automate Developer Onboarding** — New developers should go from zero to running the application locally in under 30 minutes. Provide a single setup script (`make setup`, `./scripts/bootstrap.sh`) that installs dependencies, configures the local environment, seeds the database, and runs a verification test. Document prerequisites clearly (OS, runtime versions, required accounts). Use devcontainers or Nix flakes for reproducible environments. Test the onboarding flow monthly by having a fresh team member follow it.
5. **Design Internal Developer Platforms (IDP)** — Build an IDP that abstracts infrastructure complexity: self-service environment creation (dev, staging), one-click service deployment, database provisioning, secret management, log access, and monitoring dashboards. Use Backstage, Port, or a custom portal. Define golden paths — opinionated, well-supported workflows for common tasks (create a new service, add a database, set up CI/CD). Golden paths should cover 80% of use cases with zero tickets filed.
6. **Craft CLI Tools with Excellent UX** — Design CLI tools with: clear help text (`--help` on every command and subcommand), consistent flag naming across subcommands, colored output with `--no-color` option, progress indicators for long operations, `--json` output mode for scripting, meaningful exit codes, tab completion scripts, and graceful error messages that suggest next steps. Follow the conventions of your ecosystem (POSIX flags, Go-style subcommands). Test CLI UX by watching users try it for the first time.
7. **Write Error Messages That Help Users Fix the Problem** — Every error message should contain: what went wrong (specific, not generic), why it happened (root cause if known), and how to fix it (actionable next step). Bad: "Invalid configuration." Good: "Missing required field 'api_key' in config.toml. Set your API key with: vibecli config set api_key YOUR_KEY, or export VIBE_API_KEY=YOUR_KEY." Include error codes for searchable documentation. Link to relevant docs or troubleshooting guides in error output.
8. **Manage Changelogs and Migration Guides** — Publish a changelog with every release following Keep a Changelog format: Added, Changed, Deprecated, Removed, Fixed, Security. For breaking changes, publish a detailed migration guide with before/after code examples and a timeline (deprecation warning in version N, removal in version N+2). Automate changelog generation from conventional commits but always review for clarity. Notify users of breaking changes via email, blog post, and in-tool deprecation warnings.
9. **Version APIs with a Clear Strategy** — Use URL-based versioning (`/v1/`, `/v2/`) for public APIs — it is explicit and cacheable. Use header-based versioning (`API-Version: 2024-01-15`) for date-versioned APIs (Stripe model). Never make breaking changes within a version. Support at least N-1 versions concurrently. Provide a sunset timeline (minimum 12 months for public APIs). Track per-version usage to know when old versions can be retired. Emit deprecation headers (`Sunset`, `Deprecation`) on old versions.
10. **Establish Developer Feedback Loops** — Create structured channels for developer feedback: quarterly developer surveys (Developer Experience Index), office hours with platform team, public roadmap with voting, Slack channel for real-time questions, and a feedback form embedded in documentation pages. Track key DX metrics: time to first API call, time to first deployment, support ticket volume by category, and documentation page bounce rates. Act visibly on feedback to build trust.
11. **Provide Self-Service Infrastructure** — Enable developers to provision infrastructure without filing tickets: databases, caches, queues, storage buckets, DNS records, and SSL certificates. Use Terraform modules, Pulumi components, or Crossplane compositions exposed through a portal or CLI. Enforce guardrails (naming conventions, size limits, cost budgets) automatically rather than through approval processes. Aim for provisioning time under 5 minutes for standard resources.
12. **Define and Maintain Golden Paths** — A golden path is the recommended, well-supported way to accomplish a common task. Document golden paths for: creating a new service, setting up CI/CD, adding observability, connecting to a database, deploying to production, and handling secrets. Keep golden paths opinionated (one way, not five alternatives) and maintained (tested quarterly). Make deviating from golden paths possible but not the default — teams that follow golden paths should get the best support and tooling.
13. **Optimize the Inner Development Loop** — Minimize the time between code change and feedback: hot reload for frontend (under 1 second), incremental compilation for backend (under 10 seconds), fast unit tests (under 30 seconds for the module), and local integration test capability. Profile your build pipeline and invest in the slowest step. Use tools like Turborepo, Nx, or Bazel for monorepo build caching. Never force developers to push to CI for basic validation that could run locally.
14. **Build Reliability into Developer Tools** — Developer tools must be more reliable than the systems they manage. Implement health checks, automated testing (including CLI integration tests), backward compatibility tests, and graceful offline behavior. Version-pin all tool dependencies. Provide rollback instructions when tool upgrades fail. Monitor tool adoption and error rates. A broken developer tool blocks the entire team, so treat it with production-level rigor.
