# On-Device and Local Inference

When deploying AI models for local or air-gapped inference:

1. **Quantization Selection** — Use Q4_K_M as the default quantization for most use cases: it provides the best balance of model quality and memory footprint (roughly 4.5 bits per weight). Use Q8_0 when output quality is paramount and you have sufficient VRAM/RAM (8 bits per weight, ~2x the size of Q4_K_M). Use Q2_K only for extremely constrained environments where Q4_K_M does not fit — quality degrades noticeably. Never use Q2_K for code generation tasks; minimum Q4_K_S for acceptable code quality.
2. **Hardware Backend Selection** — Select the inference backend in this priority order: (1) CUDA (NVIDIA GPU) — highest throughput, use `llama.cpp` with `-ngl 99` to offload all layers; (2) Metal (Apple Silicon) — near-CUDA speed on Mac, enable with `-ngl 1` for full GPU offload; (3) ROCm (AMD GPU) — functional but verify driver compatibility for your GPU generation; (4) CPU with AVX2/AVX-512 — acceptable for small models (<7B), slow for large models; (5) CPU without SIMD — last resort, expect 3–10x slowdown vs AVX2.
3. **Local-Only Enforcement Use Cases** — Enforce local-only inference when: the user has explicitly enabled a privacy mode, the workspace contains files matching sensitive patterns (`.env`, `*_private.*`, files in a `confidential/` directory), regulatory requirements prohibit data egress (HIPAA, GDPR data residency), or the environment is air-gapped with no internet access. Local enforcement must block all API calls to cloud providers, not just route to local — implement at the network layer or via a local-only proxy.
4. **Model Benchmarking Before Deployment** — Before deploying a quantized model to users, benchmark on a task-representative test set: measure tokens/second (generation speed), time-to-first-token (latency), and quality score (ROUGE/code eval metric vs the full-precision reference model). Document benchmark results in the model's metadata. Reject any quantized model that scores more than 5% below the full-precision baseline on the primary task metric.
5. **VRAM Estimation** — Estimate VRAM requirements before loading: `VRAM_GB ≈ (params_B × bits_per_weight / 8) × 1.15` (15% overhead for KV cache and activations at default context length). Examples: 7B Q4_K_M ≈ 4.5GB, 13B Q4_K_M ≈ 8.4GB, 70B Q4_K_M ≈ 42GB. Add 2GB per 4K tokens of context length increase beyond the default. If VRAM is insufficient, automatically select the next lighter quantization tier that fits rather than crashing.
6. **Model Download and Integrity Verification** — Always verify GGUF file integrity with SHA-256 after download. Store the expected hash in a manifest file alongside the model. Refuse to load a model whose hash does not match the manifest. Download models from a configured trusted registry (default: HuggingFace with pinned commit hash) rather than arbitrary URLs. Cache downloaded models in a content-addressed local store to avoid re-downloading unchanged models.
7. **Thermal and Power Management** — On mobile and laptop devices, monitor CPU/GPU temperature during inference. If temperature exceeds 90°C, reduce the number of threads (CPU) or offload layers back to CPU from GPU (llama.cpp `-ngl` reduction). Implement a power-saver mode that halves generation speed to stay within a configurable TDP budget. Surface thermal throttling events to the user so they understand the performance degradation.
8. **Fallback to Cloud on Capability Gap** — When the local model cannot fulfill the request (context too long, task requires capabilities not in the local model, or quality threshold not met after retry), offer a fallback to cloud with explicit user consent. Never silently switch from local to cloud — the user chose local for a reason (privacy, cost, or connectivity). Display the specific reason the local model could not fulfill the request alongside the fallback offer.
