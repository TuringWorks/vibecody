# Visual UI Feedback and Design Mode for Agents

When providing visual UI feedback to agents via screenshots and annotations:

1. **Annotation Types and When to Use Each** — Use five annotation types: (1) `ARROW` — point to a specific element that needs attention; (2) `BOUNDING_BOX` — highlight a region with a label for spatial context; (3) `TEXT_OVERLAY` — add a floating text note near an element; (4) `REDLINE` — mark spacing, alignment, or sizing issues with measurement lines; (5) `CALLOUT` — connect a distant label to a target with a leader line. Choose the type that makes the relationship between annotation and target unambiguous with no additional text.
2. **Priority Ordering** — Assign each annotation a priority: 1 = critical (blocks ship, accessibility violation, broken layout), 2 = high (significant UX problem, wrong color token), 3 = medium (style inconsistency, minor alignment), 4 = low (polish, nice-to-have). Always include the priority number in the annotation label. Agents must address all priority-1 annotations before any priority-2. Never include more than 5 priority-1 annotations per screenshot — if more issues exist, split into multiple feedback passes.
3. **Converting Annotations to Unambiguous Instructions** — Each annotation must map to exactly one actionable instruction in the accompanying text: "Change the button background from `#FF0000` to `--color-primary`", not "fix the button color". Instructions must reference: the component name (or CSS selector if unnamed), the property to change, the current value, and the target value. Vague instructions ("make it look better") are not acceptable in agent feedback loops.
4. **Design Token Extraction and Reference** — Before generating feedback, extract the active design token set from the codebase (`design-system/tokens.css`, `tailwind.config.js`, or equivalent). When an annotation identifies a color, spacing, or typography issue, reference the correct design token by name in the instruction, not a raw hex/px value. This ensures the fix is consistent with the design system rather than introducing one-off values.
5. **Screenshot Diff Annotations** — When providing iterative feedback (before/after), generate a diff annotation layer showing what changed between screenshot versions. Highlight resolved annotations in green, unresolved in red, and regressions (newly broken) in orange. Include a summary count: "3 resolved, 1 unresolved, 1 regression" at the top of the diff view. This prevents re-annotating already-fixed issues.
6. **Coordinate System Normalization** — Always specify annotation positions in normalized coordinates (0.0–1.0 relative to screenshot dimensions) rather than pixel offsets, so annotations remain valid when screenshots are resized or the display scale changes. When exporting annotations to JSON for agent consumption, include both normalized coordinates and the original pixel dimensions for reference.
7. **Agent-Readable Annotation Format** — Export annotations as structured JSON: `{"id": "ann_001", "type": "BOUNDING_BOX", "priority": 1, "target_selector": ".submit-btn", "position": {"x": 0.45, "y": 0.72, "w": 0.1, "h": 0.05}, "instruction": "Replace background with var(--color-primary)"}`. The agent processes this JSON directly alongside the screenshot; do not rely on the agent parsing annotation text from the image pixel content.
8. **Review Loop Completion Criteria** — A design mode review pass is complete when: all priority-1 annotations are resolved, no new priority-1 regressions are introduced, and the latest screenshot diff shows green for all previously annotated regions. Do not request a new screenshot until the agent confirms all outstanding priority-1 instructions have been applied — premature re-screening wastes annotation effort.
