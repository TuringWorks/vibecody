# Project Scheduling & Resource Planning

When working with project scheduling and resource management:

1. Create a comprehensive Work Breakdown Structure (WBS) as the foundation of the schedule — decompose the project scope into manageable work packages using a hierarchical structure (Level 1: project, Level 2: phases or deliverables, Level 3: work packages, Level 4: activities), apply the 100% rule (the WBS must capture 100% of the project scope, no more and no less), define work packages at a level where they can be reliably estimated (typically 8-80 hours of effort or 1-2 reporting periods in duration), use noun phrases for deliverable-oriented WBS elements (not verb phrases which describe activities), assign a unique WBS code to each element for traceability, involve the project team and subject matter experts in WBS development, and create a WBS dictionary that describes the scope, assumptions, and acceptance criteria for each work package.

2. Apply Critical Path Method (CPM) to identify schedule constraints — build the project network by defining all activities and their logical dependencies (predecessors and successors), calculate early start (ES), early finish (EF), late start (LS), and late finish (LF) for each activity through forward and backward pass, compute total float (LS - ES or LF - EF) for each activity, identify the critical path as the longest path through the network (activities with zero total float), recognize that the critical path determines the minimum project duration and any delay on it directly extends the project, monitor near-critical paths (activities with small float values) as potential risks to the schedule, and update the critical path analysis whenever the schedule changes to identify shifts in criticality.

3. Use PERT estimation for activities with significant uncertainty — gather three-point estimates for each uncertain activity: optimistic (O, best case with everything going well), most likely (M, most probable duration based on experience), and pessimistic (P, worst case considering what could go wrong), calculate the expected duration using the PERT formula: tE = (O + 4M + P) / 6, calculate standard deviation for each activity: sigma = (P - O) / 6, estimate the probability of completing the project by a target date using the sum of variances along the critical path and the normal distribution (z-score), use PERT estimates during the planning phase when historical data is limited, and transition to deterministic estimates as the project progresses and uncertainty decreases.

4. Build effective Gantt charts for schedule communication — display activities as horizontal bars with start and finish dates on a calendar timescale, show dependencies between activities using link lines (finish-to-start is most common), color-code bars to distinguish phases, responsible parties, or status (on track, delayed, complete), include milestones as diamond markers at key decision points and deliverable deadlines, display the critical path visually (bold or red bars) to highlight schedule drivers, add a progress indicator (percent complete) to each activity bar, use summary bars to roll up detailed activities into phases for executive reporting, and include a data date (status date) vertical line to separate actual progress from forecast.

5. Perform resource leveling and allocation to create achievable schedules — identify resource requirements for each activity (labor hours by skill, equipment, materials), detect resource over-allocations where demand exceeds availability using resource histograms, resolve over-allocations by leveling: delay non-critical activities within their float to spread resource demand, split activities to allow resources to work on higher-priority tasks, or assign alternative resources with equivalent capability. Distinguish between resource leveling (delay activities to eliminate over-allocation, potentially extending the project) and resource smoothing (optimize resource usage without extending the project, only using available float), maintain a resource calendar for each resource type (accounting for vacations, holidays, part-time availability), and plan for ramp-up time when onboarding new team members.

6. Define and track milestones as key schedule control points — establish milestones at major project events: project kickoff, requirements freeze, design completion, prototype delivery, testing complete, customer acceptance, and project closeout, assign milestones zero duration (they mark a point in time, not a span), link milestones to contractual obligations and payment terms, use milestone charts for executive and stakeholder reporting (simpler than full Gantt charts), track milestone achievement dates against baseline dates to measure schedule performance, implement milestone toll gates where stakeholder approval is required before proceeding, and escalate milestone slippage immediately with a recovery plan.

7. Manage dependencies correctly to maintain schedule logic — use the four standard dependency types: Finish-to-Start (FS, most common — successor cannot start until predecessor finishes), Start-to-Start (SS — activities start concurrently), Finish-to-Finish (FF — activities finish concurrently), and Start-to-Finish (SF — rare, used mainly for just-in-time constraints), apply leads (negative lag, allowing overlap) cautiously as they can mask risk and create unrealistic schedules, use lags (positive delay, such as curing time for concrete) with documented justification, avoid discretionary dependencies where possible to maintain scheduling flexibility, distinguish between mandatory dependencies (hard logic driven by the nature of the work) and discretionary dependencies (preferred sequence based on best practices), and never use constraints (Must Start On, Must Finish On) as substitutes for proper logic — use them only for externally imposed dates.

8. Implement Earned Value Management (EVM) for schedule and cost performance measurement — establish a Performance Measurement Baseline (PMB) from the approved schedule and budget, calculate key EVM metrics at each reporting period: Planned Value (PV, budgeted cost of work scheduled), Earned Value (EV, budgeted cost of work performed), and Actual Cost (AC, actual cost of work performed), derive schedule performance: Schedule Variance (SV = EV - PV, negative means behind schedule) and Schedule Performance Index (SPI = EV / PV, below 1.0 means behind schedule), derive cost performance: Cost Variance (CV = EV - AC) and Cost Performance Index (CPI = EV / AC), forecast completion using Estimate at Completion (EAC = BAC / CPI for cost trends, or calculate independently using bottom-up estimate-to-complete), and present EVM data in S-curves showing PV, EV, and AC trends over time.

9. Use Microsoft Project effectively for schedule management — set the project calendar and resource calendars before entering activities, enter activities in outline format matching the WBS, link activities with dependencies (avoid date constraints unless externally imposed), assign resources and set work, duration, and units correctly (understand how MS Project calculates the scheduling formula: Work = Duration x Units), use the Task Inspector to diagnose scheduling issues, set the baseline before execution begins (Baseline → Set Baseline), update progress by entering actual start, actual finish, and remaining duration or percent complete, use custom fields and filters to create focused views (critical activities, overdue tasks, activities by owner), and generate reports (Gantt with progress, resource usage, milestone report) for stakeholder communication.

10. Leverage Oracle Primavera P6 for large and complex project portfolios — organize the Enterprise Project Structure (EPS) to reflect the organizational hierarchy, create the project WBS before adding activities, assign activity codes for multi-dimensional grouping and filtering (by phase, area, discipline, responsibility), define calendars at the global, project, and resource levels, use Primavera's built-in resource optimization and leveling engine for large resource pools, establish a baseline schedule and store it for comparison, use scheduling options (retained logic vs. progress override) deliberately and document the choice, run the schedule check (F9 with diagnostics) to identify logic gaps and constraint issues, use Primavera's claim digger and schedule comparison features for delay analysis, and maintain the schedule in a multi-user database environment with appropriate access controls and approval workflows.

11. Apply schedule compression techniques when the deadline is at risk — use crashing to reduce duration of critical-path activities by adding resources: identify activities where adding resources actually reduces duration (not all tasks are parallelizable), calculate the crash cost per day saved for each critical activity and crash the cheapest first, monitor for new critical paths that emerge as you crash activities. Use fast-tracking to overlap activities that were originally planned sequentially: identify FS dependencies that can be converted to SS with a lead, assess the rework risk of starting downstream activities before upstream activities are fully complete, and plan additional reviews and coordination to manage the increased risk. Consider scope reduction, phased delivery, or deadline negotiation as alternatives when crashing and fast-tracking create unacceptable cost or risk.

12. Report schedule progress with clear baselines and variance analysis — compare current schedule against the approved baseline at every reporting period, calculate schedule variance for each activity and the overall project (planned vs. actual dates), present progress using a Gantt chart with baseline bars shown alongside current bars, summarize status with a traffic-light dashboard (green: on track within tolerance, yellow: at risk with mitigation plan, red: behind schedule requiring management action), explain the root cause for every significant variance (not just the fact of delay), present the recovery plan with specific actions, responsible parties, and revised forecast dates, forecast the project completion date based on current trends (not just the original plan), and maintain a schedule change log documenting all approved changes to the baseline with justification.
