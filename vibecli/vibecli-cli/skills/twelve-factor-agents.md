# Twelve-Factor Agents

When building AI agents following the 12-Factor Agents methodology:

1. **Natural Language to Tool Calls** — Transform natural language inputs into structured tool calls rather than generating free-form text that gets parsed downstream. Define a clear tool schema (JSON Schema or similar) for every action, validate inputs before execution, and treat the LLM as a translation layer that converts intent into deterministic, auditable function invocations.
2. **Own Your Prompts** — Control your system prompts, few-shot examples, and instruction templates directly in your codebase rather than relying on framework-generated defaults. Version-control prompts alongside code, A/B test prompt variations, and treat prompt engineering as a first-class engineering discipline with review processes and regression tests.
3. **Own Your Context Window** — Actively manage what goes into the context window instead of naively stuffing everything in. Implement strategic summarization, relevance filtering, and recency-based pruning. Track token budgets, reserve space for tool outputs, and use retrieval-augmented generation to pull in only the most relevant context on demand.
4. **Tools Are Just Structured Outputs** — Treat tool calls as structured data outputs from the model, not as magical execution primitives. Parse tool call responses with the same rigor as any API contract: validate schemas, handle missing fields, apply type coercion, and maintain a registry of tool definitions that serves as both documentation and runtime validation.
5. **Unify Execution State and Business State** — Merge your agent's workflow execution state (current step, pending tool calls, retry count) with your business domain state (order status, user preferences) into a single persistent record. This eliminates synchronization bugs, makes debugging trivial, and lets you query agent progress using the same data layer as your application.
6. **Launch/Pause/Resume with Simple APIs** — Design agents that can be started, suspended, and resumed through simple API calls (start, pause, resume, cancel). Serialize the full agent state including conversation history, pending actions, and execution pointer so that an agent can be frozen mid-task, moved to a different node, and resumed without loss of context or duplicated work.
7. **Contact Humans with Tool Calls** — Implement human-in-the-loop as a standard tool call (e.g., `ask_human`, `request_approval`) rather than special-casing it in the control flow. The agent requests human input the same way it requests any other resource, and the orchestrator routes it to the appropriate channel (Slack, email, UI modal) while the agent waits asynchronously.
8. **Own Your Control Flow** — Maintain explicit control over the agent's decision-making loop rather than delegating it entirely to the LLM. Use deterministic routing for known patterns, constrain which tools are available at each step, enforce maximum iteration limits, and implement hard guardrails that cannot be overridden by model outputs.
9. **Compact Errors into Context Window** — When tool calls fail, compress error information into a concise, structured format before feeding it back into the context window. Include the error type, a brief message, and suggested remediation rather than raw stack traces. This preserves context budget while giving the model enough signal to self-correct on the next attempt.
10. **Small, Focused Agents** — Build specialized agents that excel at specific tasks rather than monolithic agents that attempt everything. A code-review agent, a deployment agent, and a data-analysis agent each with tailored prompts, tools, and guardrails will outperform a single general-purpose agent. Compose them via orchestration when complex workflows require multiple capabilities.
11. **Trigger from Anywhere** — Enable agent activation from any channel: CLI commands, HTTP webhooks, Slack messages, cron schedules, CI/CD events, or other agents. Decouple the trigger mechanism from the agent logic by accepting a standardized input envelope. This makes agents composable and lets users interact through whatever interface fits their workflow.
12. **Make Your Agent a Stateless Reducer** — Design each agent step as a pure function: given the current state and a new event (user message, tool result, error), produce the next state and a list of side effects. This makes agents deterministic, testable with unit tests, replayable from logs, and horizontally scalable since any worker can process any agent step without shared mutable state.
