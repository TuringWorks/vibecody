# Autonomous Deployment Pipelines

When building or operating agent-driven autonomous deployment pipelines:

1. **Always Include Health Gates** — Every deployment pipeline must include post-deployment health gates before marking a deployment as successful. Minimum gates: HTTP health endpoint returns 200, error rate is below baseline (default threshold: <0.1%), p99 latency is within 20% of baseline, and at least one synthetic smoke test passes. Health gates must have a configurable observation window (default: 5 minutes) before the gate is declared passed. A deployment is not complete until all gates pass.
2. **Dry-Run Before First Real Deploy** — On first-ever deployment to any environment, run a dry-run pass that logs all intended actions (container image to deploy, config values to set, secrets to inject, DNS changes to apply) without executing them. Present the dry-run output to the user for review. Require explicit confirmation before proceeding to live execution. For subsequent deploys to the same environment, dry-run is optional but should be triggered automatically when the diff is unusually large (>20 file changes).
3. **Rollback Trigger Thresholds** — Define automatic rollback triggers: error rate exceeds 1% for >2 minutes, p99 latency exceeds 3× baseline for >2 minutes, health endpoint returns non-200 for >30 seconds, or any critical alert fires within the observation window. Rollback must complete within 5 minutes of trigger. Log the trigger condition, rollback start time, and rollback completion time. Never suppress rollback triggers — if automatic rollback fails, escalate to on-call immediately.
4. **Staging to Production Requires Gate Pass** — Promotion from staging to production is gated on: all staging health gates passing, staging load test passing (p99 <SLA, error rate <0.1%), security scan returning no new critical findings, and a minimum soak time in staging (default: 30 minutes). An agent may perform the promotion automatically only after all gates pass. If any gate fails, halt promotion and notify the user; never promote automatically on a failed gate even if the failure appears transient.
5. **Never Skip the Build Stage** — The build stage (compile, test, lint, security scan) must always run before any deployment, regardless of urgency. Do not deploy from a local working directory, an untagged image, or an artifact that was not produced by a clean CI build. If a deployment is needed urgently and the build is slow, optimize the build — do not skip it. Skipping the build stage introduces untested, potentially vulnerable artifacts into production.
6. **Immutable Artifact Versioning** — Every deployable artifact must have a unique, immutable version tag (content hash or semantic version + build number). Never redeploy by pushing a new artifact under an existing tag (e.g., `latest`). The deployment record must store the exact artifact tag deployed so rollbacks always target a specific known-good artifact. Mutable tags make rollbacks non-deterministic.
7. **Environment Parity Enforcement** — Staging must be as close to production as possible: same container image, same config schema (with environment-specific values), same infrastructure topology (scaled down is acceptable, different topology is not). An agent that detects parity drift (staging running a different image tag than the one being tested) must halt the pipeline and surface the drift to the user before proceeding.
8. **Deployment Blast Radius Control** — For production deployments, use progressive rollout strategies (canary, blue/green, or ring-based) to limit blast radius. Default: canary to 5% of traffic for the observation window, then promote to 100% if health gates pass. An agent managing the rollout must monitor the canary segment independently from the stable segment and trigger rollback only for the canary (not a full rollback) if canary-specific health metrics degrade.
