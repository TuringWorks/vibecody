# LLM Engineering Best Practices

When working with LLM engineering:

1. Select models based on the size-quality-cost tradeoff for your use case — use large frontier models (Claude, GPT-4) for complex reasoning and code generation, mid-size models (Llama 70B, Mixtral) for balanced quality and cost, and small models (Phi, Gemma 2B-7B) for high-throughput low-latency tasks like classification and extraction; benchmark on your actual task distribution before committing.
2. Apply quantization to reduce model size and inference cost — use GGUF format with llama.cpp for CPU/mixed inference, GPTQ for GPU-optimized 4-bit quantization with minimal quality loss, and AWQ (Activation-aware Weight Quantization) for the best quality-compression tradeoff; always evaluate perplexity and task-specific accuracy after quantization to verify acceptable degradation.
3. Deploy models with production-grade serving frameworks — use vLLM for high-throughput GPU serving with PagedAttention and continuous batching, Text Generation Inference (TGI) for Hugging Face model compatibility with tensor parallelism, and Ollama for local development and single-machine deployment; configure health checks, graceful shutdown, and request queuing for all production deployments.
4. Fine-tune models efficiently with LoRA (Low-Rank Adaptation) or QLoRA for memory-constrained environments — prepare high-quality training data (1K-10K curated examples outperform 100K noisy ones), use a rank of 8-64 based on task complexity, train for 1-3 epochs to avoid overfitting, and merge adapters for deployment to eliminate inference overhead.
5. Evaluate model performance systematically using frameworks like lm-eval-harness for standardized benchmarks and custom evaluation suites for domain-specific tasks — measure accuracy, hallucination rate, instruction following, and safety; run evaluations on every model change, fine-tune iteration, and prompt revision to track regressions.
6. Optimize inference with prompt caching and KV cache management — enable prefix caching in vLLM to reuse KV cache across requests sharing system prompts, set appropriate `max_model_len` to bound memory usage, and use sliding window attention models for long-context workloads; monitor cache hit rates and adjust system prompt design to maximize prefix sharing.
7. Implement batching strategies to maximize throughput — use continuous batching (vLLM, TGI) over static batching for variable-length sequences, configure `max_batch_size` and `max_waiting_tokens` based on latency SLAs, and separate batch processing workloads from real-time serving with dedicated inference pools to prevent queue starvation.
8. Manage GPU memory carefully across the inference stack — profile VRAM usage with `nvidia-smi` and framework-specific tools, use tensor parallelism across multiple GPUs for models exceeding single-GPU memory, configure `gpu_memory_utilization` (vLLM default 0.9) to leave headroom for request bursts, and implement request-level memory limits to prevent OOM crashes from long-context inputs.
9. Build model routing and fallback systems for resilience — implement a router that directs simple queries to smaller/cheaper models and complex queries to capable models, configure automatic fallback chains (primary -> secondary -> tertiary provider), use circuit breakers with health checks per model endpoint, and track per-model latency and error rates to inform routing decisions dynamically.
10. Deploy guardrails and safety filters as a separate layer — use input classifiers to detect prompt injection, jailbreak attempts, and PII before reaching the model; apply output filters for toxicity, factuality checks, and format validation; implement rate limiting per user/API key; and log all filtered requests for security review without blocking the inference pipeline.
11. Instrument comprehensive observability for LLM operations — track request latency (time-to-first-token and total generation time), tokens per second, input/output token counts, cost per request, cache hit rates, and error rates; use structured logging with request IDs for tracing, set up dashboards in Grafana/Datadog, and alert on latency P99 degradation and error rate spikes.
12. A/B test model versions and prompt changes rigorously — deploy new models behind feature flags with percentage-based traffic splitting, collect user feedback signals (thumbs up/down, task completion rate, edit distance), run statistical significance tests before full rollout, and maintain the ability to instant-rollback to the previous version; never ship a model change without measurable comparison data.
